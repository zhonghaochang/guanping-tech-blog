多模态 DeepSearch 的训练路线：四篇工作的数据、工具与 Harness

我想弄清楚的不是哪篇论文分数最高，而是一个只会搜图、搜文字的模型，怎样一步步变成长轨迹、多工具的研究 Agent。

常中豪2026 年 7 月 12 日

最近几篇多模态 DeepSearch 论文看起来很像：都是视觉语言模型，都会图片搜索和文本搜索，也都用到了 SFT 或 RL。把实验表放在一起却很难比较，因为它们训练的并不是同一种 Agent。

MMSearch-R1 的环境只有两种搜索工具，轨迹很短，主要研究“模型能不能自己决定要不要搜”。到了 OpenSearch-VL，动作空间已经包含裁剪、OCR、超分、透视矫正和两种检索，单条 rollout 可以有 7 万个 response tokens。中间的 DeepMMSearch-R1 和 Vision-DeepResearch，刚好把这条演进路线连接起来。

所以这篇文章不按发布时间罗列摘要，而是沿着三个问题读：训练数据怎样从问题变成轨迹，工具环境扩大以后模型怎样冷启动，以及长轨迹如何不被真实搜索服务拖垮。

先把四篇工作放在一张表里

工作它主要解决什么训练数据Agent 能做什么
MMSearch-R1不用 SFT，RL 能否学会按需搜索约 5K RL；另有 8K SFT 对照图片搜索、文本搜索；短轨迹
DeepMMSearch-R1先用专家轨迹教会局部图搜和多轮查询10K SFT + 约 5K RL整图/裁剪图搜索，多次文本搜索
Vision-DeepResearch把视觉定位接到长程文本研究能力上30K SFT + 15K RL多尺度裁剪、视觉检索、网页访问、Python
OpenSearch-VL开放多工具训练配方，并处理长轨迹失败36,592 SFT + 8K RL7 种工具，主动改善图片再检索

这张表里最重要的不是样本数从 5K 涨到 36K，而是 action space 变了。动作只有“搜图还是搜文字”时，RL 很容易探索；动作变成七种之后，模型连第一步应该裁剪还是 OCR 都要学，冷启动的难度完全不同。

MMSearch-R1：先回答“什么时候应该搜”

MMSearch-R1 用 Qwen2.5-VL-7B-Instruct 做基础模型，主实验没有先做 SFT，直接在真实图片搜索和文本搜索环境里跑 GRPO。它的设计很克制：图片搜索只能放在前面，正文描述每条 rollout 最多 3 个对话轮次、最多 2 次搜索，然后必须回答。

数据构造的几个数字容易混在一起。作者先随机取 10,000 个视觉概念，自动生成 6,000 条 visual-concept VQA，其中 5,400 条训练、600 条测试；再加入 7,000 条 InfoSeek 文本知识样本和 800 条人工样本。经过 search balancing 后，真正用于 RL 的 FVQA-train 约 5,000 条，其中 3,400 条需要搜索，1,600 条不需要搜索。

“不需要搜索”的 1,600 条并不是凑数。训练集如果全是 search-required，模型最容易学到的策略就是每题都搜。MMSearch-R1 的 reward 也在约束这件事：答对是主信号，答对但使用搜索会被施加搜索惩罚，格式 reward 权重为 0.1。

MMSearch-R1 的短协议
<reason>图像里没有足够信息，需要先确认实体。</reason>
<search><img></search>

<information>前 5 个图片匹配与网页标题……</information>

<reason>实体已确定，还需要核验属性。</reason>
<text_search>entity attribute</text_search>

<answer>最终答案</answer>

官方实现里，图片检索和文本检索都以 SerpAPI 为入口；文本结果再通过 Jina Reader 读取网页，由 Qwen3-32B 压成和 query 有关的短摘要。RL 每个 prompt 采样 8 条轨迹，batch 512，在 4×8 张 H100 上训练，学习率 2e-6，KL 系数 0.001，clip ratio 0.2。

论文另外做了一个 SFT 对照：GPT-4o 生成 13.2K 条轨迹，过滤掉 5.2K 条错答后剩 8K。这个对照有意思的地方是，RL 的 5K 并不比 SFT 的 8K 大，却学到了更节制的搜索行为。这里比较的其实是“模仿一条既定路线”和“在环境里试出路线”的区别。

DeepMMSearch-R1：为什么又把 SFT 加了回来

DeepMMSearch-R1 还是 Qwen2.5-VL-7B-Instruct，但搜索动作多了一层：模型可以搜整张图，也可以先用 referring expression 指定局部实体。Harness 用 GroundingDINO 找到 bounding box，裁剪以后再做图片搜索。模型不只决定“搜不搜”，还要决定“搜哪一块”。

这时直接靠 RL 探索就贵了。作者先从 InfoSeek 随机取 200,000 个问题，让 Gemini-2.5-Pro 在真实工具环境里生成多轮轨迹。最终答案必须和 ground truth 一致，才会留下；这一步剩约 47,000 条。再按知识类型平衡成 10,000 条 SFT 数据，search-required 和 search-free 各占一半。

这 10K 并不都是长链。1 到 6 轮的样本数分别是 5000、3289、1455、184、58、14；工具调用中，文本搜索占 68.9%，整图搜索 13.5%，裁剪图搜索 17.6%。从分布上看，SFT 主要在教模型“第一步别选错”，而不是让它模仿几十轮研究。

SFT 只在语言模型上加 LoRA（rank 8），视觉编码器和 projector 都冻结；训练 3 epochs，学习率 1e-4，使用 1×8 张 H100，global batch 8。工具返回不计算 loss。RL 阶段允许一次图片或裁剪图搜索、多个文本搜索，总工具调用不超过 10，每题采样 8 条 rollout。

复现时需要注意一个论文内部的不一致：正文写 RL 训练 30 epochs、batch 512；附录写 20 epochs、batch 256。两处都写了 4×8 张 H100、8 rollouts、45-step warmup。文章和附录都能在当前 PDF 中找到，我没有替作者选其中一组。

Vision-DeepResearch：轨迹一长，问题就不只在模型里

Vision-DeepResearch 把基础模型换成 Qwen3-VL-8B-Instruct 和 Qwen3-VL-30B-A3B-Instruct。它没有继续给图片搜索增加更多标签，而是想办法把视觉阶段接到已经成熟的文本 DeepResearch 能力上。

数据生成先让多模态模型在图片上做多实体、多尺度裁剪和搜索。找到候选页面后，辅助模型把长网页总结成和当前图片区域有关的证据。视觉实体基本确定以后，再把图片写成详细描述，交给文本 DeepResearch 模型继续网页搜索、访问、总结和 Python 推理。

视觉阶段怎样接到文本研究阶段
visual_state = {
    "question": question,
    "image_description": describe(original_image),
    "identified_entities": ["..."],
    "visual_evidence": ["crop_2 matched page_x", "..."],
}

text_agent.run(
    state=visual_state,
    tools=[web_search, visit_page, summarize_page, python],
)

SFT 一共 30K：16K verified factual VQA 轨迹、8K text-only QA 轨迹、6K fuzzy VQA 轨迹。加入 text-only 数据并不是偏题，它在补长程搜索和网页阅读，而视觉前缀负责告诉文本 Agent 当前实体和证据是什么。RL 使用 15K VQA，其中 10K 来自过滤后的现有数据，5K 是 fuzzy VQA。

这一阶段，单条 rollout 可以到 50 轮，context 64K，每轮 response 最多 4K。同步 rollout 会让所有 GPU 等最慢的网页请求，所以论文使用队列式异步调度器和工具池，让生成、搜索和网页访问错开执行，并报告了超过 10 倍的吞吐提升。

长轨迹也引入了很实际的坏状态：模型重复生成、工具格式连续错误、上下文耗尽。论文用 n-gram 检测重复；连续 3 次格式或工具错误就终止。异常轨迹不参与反向传播，但仍留在组内 advantage 计算里。到这里，Harness 已经不是“给模型接几个 API”，而是训练稳定性的一部分。

这篇论文披露了数据规模、轮次和上下文设置，但没有在 PDF 中给出完整的 GPU 数量与学习率。我宁愿把这一栏留空，也不从相邻工作推一个看起来合理的数字。

OpenSearch-VL：让模型先把图片变得可搜索

OpenSearch-VL 的变化最直观：工具从“找外部知识”扩展到“改善当前观察”。它提供 7 种工具：

工具实现模型为什么会用它
TextSearchSerper → Jina Reader → Qwen3-32B 摘要查实体属性和网页证据
ImageSearchPolaris Lens API从图片定位实体和相关页面
Crop本地确定性裁剪把注意力放到局部目标
OCRPaddleX 版面解析读取标牌和文档文字
SharpenOpenCV unsharp mask处理轻微模糊
SuperResolutionEDSR放大小目标
PerspectiveCorrect边缘检测与四点变换校正倾斜招牌或页面

这一套工具要求 Harness 维护 active visual context。原图经过裁剪、超分和透视矫正后，每张派生图都要有稳定 ID 和父子关系；否则后面的 OCR 不知道应该读哪张图。

active_visual_context.json
[
  {"image_id": "img_0", "parent": null, "path": "question.jpg"},
  {"image_id": "img_1", "parent": "img_0",
   "operation": {"name": "Crop", "box": [420, 90, 980, 530]}},
  {"image_id": "img_2", "parent": "img_1",
   "operation": {"name": "SuperResolution", "scale": 4}}
]

训练数据也专门为这些工具服务。问题来自 Wikipedia 实体路径以及 LiveVQA、FVQA、WebQA，先过滤无需工具就能答对、或一次图片搜索就能答对的题；再把 10% 图片做模糊、降采样和透视变形。Claude Opus 4.6 在真实环境里每题采样 5 条专家轨迹，GPT-4o 判断答案，GPT-5.4 判断过程，最后得到 36,592 条 SFT 轨迹，平均 6.3 次工具调用。

SFT 是全参数训练，视觉塔和 projector 都不冻结：32K cutoff、8 epochs、学习率 2e-5、effective batch 256，使用 256 张 H20。RL 数据是和 SFT 不重叠的 8K VQA；64 张 H20 上运行异步 rollout，8B 每题采样 8 条，30B 每题 16 条，最大 response 长度 70,000 tokens，同时允许 256 个任务和 2,048 个工具调用并行。

真实工具环境最棘手的不是某一次搜索为空，而是错误会沿着轨迹传播。OpenSearch-VL 把连续 3 次工具错误定义为 fatal：fatal 之后的 token 不计算策略梯度；fatal 之前的有效 prefix 仍留着，并对负 advantage 做单边截断。这样既不必丢掉整条长轨迹，也不会让模型学习错误后的随机输出。

把这条路线重新画一遍

读完四篇论文，我觉得路线并不是简单的“工具越来越多、上下文越来越长”，而是下面四步：

  1. 先让模型学会按需搜索。MMSearch-R1 用 search-balanced 数据和搜索惩罚，解决“每题都搜”的坏习惯。
  2. 再把视觉定位变成一个动作。DeepMMSearch-R1 加入裁剪图搜索，SFT 用来降低更大动作空间的探索成本。
  3. 把视觉前缀接到长程文本研究。Vision-DeepResearch 借用文本 Agent 的网页搜索和推理能力，重点转向异步 Harness。
  4. 让模型主动修复观察。OpenSearch-VL 加入图像增强和 OCR，同时用 visual state 与 fatal-aware RL 管住长轨迹。

这也解释了为什么“更多 RL”不是统一答案。两工具、三轮对话的环境可以直接探索；七工具、几十轮的环境如果没有 SFT 起点和稳定 Harness，大部分算力会消耗在非法动作、服务超时和重复搜索上。

如果自己做一个最小复现

我会从 MMSearch-R1 的小环境开始，只保留图片搜索和文本搜索，并把“无需搜索”的问题明确混进训练集。等到查询、停止和日志都稳定，再加 Crop，让每张派生图有自己的 image_id。

OCR 和图像增强应该最后加。它们不仅增加工具数量，还会改变 observation 的形态。每加一个工具，都要重新检查三个指标：它是否真的带来新证据，失败后能否恢复，以及模型有没有把它当成固定前置动作。

真正进入长轨迹 RL 前，我还会先记录搜索空结果率、网页超时率、平均工具数、重复 query 比例、fatal rate 和每题成本。没有这些数据，只看最终准确率，很难分辨提升来自模型策略，还是来自某个更强、更慢的搜索后端。
