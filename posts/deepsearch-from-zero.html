一个 Agent Search 轨迹是怎么样的？

从运行时消息、SFT loss mask 到 RL rollout，逐层拆开一个搜索 Agent 的样本。重点不是“调用搜索 API”，而是模型、工具环境和训练目标怎样共同塑造一次搜索。

常中豪2026 年 7 月 12 日

如果只看最终回答，DeepSearch 和普通问答没有区别。真正需要训练的是中间过程：模型先看到了什么，为什么发起这次搜索，工具返回了什么，下一步为什么改查另一个线索，以及什么时候证据已经足够。

这篇文章从一条代表性轨迹开始，把它分别改写成运行时消息、SFT 样本和 RL rollout。示例采用“根据一张桥梁照片识别地点并确认通车年份”的任务；它是为了讲清结构而重写的工程示例，不是某篇论文逐字公开的训练数据。

1. 先把一次 DeepSearch 拆开

一条多模态 DeepSearch 轨迹至少包含五种状态：原始问题、视觉观察、模型动作、工具观察和最终答案。它不是一段越写越长的思维文本，而是模型和环境交替推进的状态机。

01Observe
问题与原图
02Act
裁剪或搜索
03Receive
候选与网页
04Verify
交叉核验
05Answer
带证据作答

例如，原图中的桥塔轮廓很明显，但路牌太小。合理轨迹不会直接猜桥名，而是先裁剪桥塔和路牌，再用图片搜索获得候选实体，最后用文本搜索确认年份。

关键区别：消息历史是模型的输入；审计轨迹是训练与评测系统保存的结构化记录。两者通常不完全相同。耗时、HTTP 状态码、原始网页和 reward 可以写进审计日志，但不必全部塞回模型上下文。

2. Harness 才是搜索 Agent 的运行时

Harness 可以理解为模型外面的控制器。它负责把工具 schema 注入上下文、解析动作、执行工具、保存新图片、压缩网页、维护预算，并把 observation 重新交给模型。模型只“提出动作”，不能直接越过 Harness 操作网络。

harness.py · 最小但完整的控制循环
from dataclasses import dataclass, field
from typing import Any

@dataclass
class Episode:
    messages: list[dict[str, Any]]
    audit: list[dict[str, Any]] = field(default_factory=list)
    calls_left: int = 8
    done: bool = False

def run_episode(model, registry, prompt, image):
    ep = Episode(messages=[{
        "role": "user",
        "content": [
            {"type": "image", "path": image},
            {"type": "text", "text": prompt},
        ],
    }])

    while not ep.done and ep.calls_left > 0:
        raw = model.generate(ep.messages, tools=registry.schemas())
        event = parse_one_event(raw)  # answer 或恰好一个 tool_call
        ep.audit.append({"model_raw": raw, "event": event})

        if event["type"] == "answer":
            ep.messages.append({"role": "assistant", "content": raw})
            ep.done = True
            break

        try:
            result = registry.execute(event["name"], event["arguments"])
            observation = normalize_observation(result)
        except Exception as exc:
            observation = {"ok": False, "error": type(exc).__name__}

        ep.messages.extend([
            {"role": "assistant", "content": raw},
            {"role": "tool", "name": event["name"],
             "content": observation},
        ])
        ep.calls_left -= 1

    return ep

这里有三个很容易被忽略的设计点。

  1. 一次只允许一个动作。这样 observation 的归因最清楚，也方便对错误之后的 token 做 mask。
  2. 工具结果必须归一化。搜索引擎原始响应会变化，训练数据应保存稳定字段，例如标题、URL、摘要和本地图片引用。
  3. 错误也是 observation。偶发超时应该让模型看到可恢复错误，而不是让整个进程崩溃；连续错误再触发终止。

3. 同一条轨迹怎样变成 SFT 样本

SFT 的作用是先教会模型协议和基本搜索策略。一个样本通常包含完整的多轮 messages；但训练 loss 并不覆盖所有 token。用户问题、工具 observation 和图片内容是条件，真正监督的是 assistant 应该输出的动作与答案。

sft_sample.json · 训练视角
{
  "messages": [
    {"role": "system", "content": "每轮输出一个 tool_call 或 final。"},
    {"role": "user", "content": [
      {"type": "image", "path": "bridge.jpg"},
      {"type": "text", "text": "桥名和正式通车年份是什么？"}
    ]},
    {"role": "assistant", "content":
      "<tool_call>{\"name\":\"crop\",\"arguments\":{\"image\":\"bridge.jpg\",\"box\":[0.18,0.12,0.82,0.88]}}</tool_call>"},
    {"role": "tool", "name": "crop", "content": [
      {"type": "image", "path": "crop_001.jpg"}
    ]},
    {"role": "assistant", "content":
      "<tool_call>{\"name\":\"image_search\",\"arguments\":{\"image\":\"crop_001.jpg\"}}</tool_call>"},
    {"role": "tool", "name": "image_search", "content":
      "候选：Kessock Bridge；Skye Bridge。"},
    {"role": "assistant", "content":
      "<tool_call>{\"name\":\"text_search\",\"arguments\":{\"query\":\"Kessock Bridge officially opened year\"}}</tool_call>"},
    {"role": "tool", "name": "text_search", "content":
      "The Kessock Bridge was opened in 1982 ..."},
    {"role": "assistant", "content":
      "<final>图中是 Kessock Bridge，1982 年正式通车。</final>"}
  ]
}
消息部分送入模型计算 token loss原因
system / tool schema是否条件，不是预测目标
user 文本与原图是否问题条件
assistant tool_call是是学习动作协议与参数
tool 文本 / 新图片是否环境 observation
assistant final是是学习停止与最终作答

实现上常用 -100 标记不参与交叉熵的 token。多模态模型还要单独处理 image token：图片参与注意力计算，但工具返回的图片不是语言模型要“预测”的目标。

collator.py · assistant-only loss mask
IGNORE_INDEX = -100

def build_labels(rendered_turns, tokenizer):
    input_ids, labels = [], []

    for turn in rendered_turns:
        ids = tokenizer(turn.text, add_special_tokens=False).input_ids
        input_ids.extend(ids)

        supervised = turn.role == "assistant"
        labels.extend(ids if supervised else [IGNORE_INDEX] * len(ids))

        # 视觉 embedding 由 processor 插入；它们只作为条件。
        for start, end in turn.image_token_spans:
            labels[start:end] = [IGNORE_INDEX] * (end - start)

    assert len(input_ids) == len(labels)
    return {"input_ids": input_ids, "labels": labels}

不同项目对“reasoning token 是否监督”处理不同：如果数据里保留了显式 reason，就会一并学习；如果只保留 tool call，模型学习到的是更短的动作策略。真正不可省的是协议边界，否则模型很容易生成两个工具调用、缺括号的 JSON，或者在 tool_call 后继续输出自然语言。

4. SFT 样本不是越长越好

高质量轨迹至少要经过四层过滤：

  • 答案验证：最终答案与标准答案或 judge 判定一致。
  • 协议验证：每个动作能被 parser 解析，参数满足 schema。
  • 过程验证：工具调用与当前证据相关，不能靠无关搜索碰巧命中。
  • 难度验证：过滤模型不用搜索就能答对的样本，否则它会学会跳过工具。

这也是为什么“生成 20 万条轨迹”和“得到 2 万条可训练轨迹”是两个概念。DeepMMSearch-R1 从 20 万个 InfoSeek 问题出发，经 Gemini 2.5 Pro 生成和答案过滤后留下约 4.7 万条，再按任务分类平衡成 1 万条 SFT 数据。样本规模必须和过滤路径一起看。

5. RL 样本和 SFT 样本到底差在哪

SFT 数据已经给出了“应该怎样走”。RL 数据通常只给初始问题和可验证答案，模型在真实工具环境中自己 rollout。环境执行每个动作，直到模型回答、预算耗尽，或者发生致命错误。

对同一个 prompt 采样一组轨迹后，组内会出现明显差异：有的直接猜错，有的搜到桥名却没核验年份，有的用三步得到正确答案，还有的十步以后才答对。GRPO 或 RLOO 这类方法利用组内相对优势更新策略。

rollout轨迹摘要正确格式工具成本总 reward
A直接猜 Skye Bridge0100.10
B图片搜索 → 文本核验 → 正确1120.94
C连续改写 7 次后正确1170.79
D产生非法 JSON，修复失败0010.00
上表数字是教学示例，不是某篇论文的原始 reward。它说明一个设计事实：只奖励答案正确，模型不会主动学习少搜、搜准和遵守协议；但工具惩罚过重，又会把策略推向“不搜索直接猜”。
reward.py · 一种可解释的组合
def reward(episode, ground_truth):
    r_answer = exact_or_judge(episode.answer, ground_truth)  # 0 或 1
    r_format = float(episode.all_events_valid)
    r_query = process_judge(episode.audit)                   # 0 到 1
    tool_cost = 0.03 * episode.successful_tool_calls

    # 无效格式直接门控；正确性仍然是主信号。
    return r_format * (0.8 * r_answer + 0.2 * r_query) - tool_cost

def group_advantages(rewards):
    mean = sum(rewards) / len(rewards)
    std = (sum((r - mean) ** 2 for r in rewards) / len(rewards)) ** 0.5
    return [(r - mean) / (std + 1e-6) for r in rewards]

6. 长轨迹最麻烦的是失败以后怎么算 loss

网页超时、图片下载失败和空检索结果属于可恢复错误。更危险的是 parser 连续失败、模型陷入重复调用，或上下文预算耗尽。此时“整条轨迹丢掉”会浪费前面有效的决策；“整条轨迹照常训练”又会把错误后的随机 token 当成有意义探索。

比较稳妥的做法是保存 fatal step：致命错误前的 prefix 仍可参与优势估计，之后的 token 不回传梯度。OpenSearch-VL 进一步对 fatal prefix 使用单边 advantage clamp，只保留正优势；Vision-DeepResearch 则把异常轨迹从反向传播中 mask，但仍留在组内优势计算中。

rollout_mask.py · 致命错误后的 token 不训练
def make_policy_mask(token_steps, fatal_step=None):
    mask = []
    for step_id, token_count in token_steps:
        trainable = fatal_step is None or step_id < fatal_step
        mask.extend([1 if trainable else 0] * token_count)
    return mask

def fatal_prefix_advantage(raw_advantage, is_fatal):
    if not is_fatal:
        return raw_advantage
    return max(raw_advantage, 0.0)

因此，一个可工作的训练顺序通常是：先让少量、高质量 SFT 把输出空间限制到合法动作；再让 RL 在真实工具环境里探索“什么时候搜、搜什么、何时停”。MMSearch-R1 的特别之处在于它展示了简单双工具环境下可以跳过冷启动 SFT，直接用 GRPO 学出搜索行为；这不等于所有复杂 Harness 都不再需要 SFT。

8. 我会怎样检查一个 DeepSearch 训练项目

  1. 先看 Harness：工具 schema、单轮动作数、图像 observation 如何回注、预算和终止条件。
  2. 再看样本：原始问题多少、生成多少轨迹、过滤后剩多少、是否过滤“无需搜索”的简单题。
  3. 确认 loss mask：user、tool 和 image token 是否被误监督。
  4. 确认 rollout：每题采样数、最大上下文、并发工具调用和失败恢复。
  5. 最后看 reward：答案信号是否压倒格式和成本，judge 是否真的能验证任务。
我的判断：DeepSearch 的核心训练对象不是“知识”，而是证据不完整时的决策策略。SFT 定义它能做什么，Harness 定义它实际做了什么，RL 决定哪些行为以后更可能再次发生。
