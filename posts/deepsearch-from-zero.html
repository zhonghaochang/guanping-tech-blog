一个 Agent Search 轨迹是怎么样的？

常中豪2026 年 7 月 12 日

“Agent 会一边思考一边搜索”这句话没什么信息量。真把系统跑起来，我们看到的也不是一大段神秘的思维过程，而是一串很朴素的事件：收到问题、生成查询、调用工具、读回结果、补下一块证据，最后提交答案。

我习惯把这串事件叫做一条 Search 轨迹。它至少要能回答三个问题：每一步为什么有必要，当前结论来自哪条证据，以及这一步结束后还缺什么。下面不讲抽象架构，直接看两条日志。

一、纯文本 Search：查清一个论文配置

先用一个真实问题：DeepMMSearch-R1 做强化学习时，batch size 到底是多少？ 这类问题看着简单，却不能只抄搜索摘要，因为论文正文和附录给了两个不同的数字。

这条轨迹实际跑了什么

text_search_trace.jsonl
{"step": 0, "event": "question",
 "text": "DeepMMSearch-R1 的 RL batch size 是多少？"}

{"step": 1, "event": "tool_call", "tool": "search_web",
 "args": {"query": "DeepMMSearch-R1 reinforcement learning batch size"},
 "decision": "先找到论文原文，不采用二手解读"}

{"step": 1, "event": "tool_result",
 "items": [{"title": "DeepMMSearch-R1", "url": "arxiv.org/abs/2510.12801"}]}

{"step": 2, "event": "tool_call", "tool": "open_document",
 "args": {"url": "https://arxiv.org/pdf/2510.12801"}}

{"step": 2, "event": "tool_result",
 "document_id": "paper_0", "pages": 24, "text_ready": true}

{"step": 3, "event": "tool_call", "tool": "find_in_document",
 "args": {"document_id": "paper_0", "query": "batch size"}}

{"step": 3, "event": "tool_result",
 "matches": [
   {"page": 5, "section": "3.3 Reinforcement Learning", "text": "batch size of 512"},
   {"page": 18, "section": "B.2 RL Training Details", "text": "batch size is set to 256"}
 ]}

{"step": 4, "event": "final",
 "answer": "论文内部存在不一致：正文第 3.3 节写 512，附录 B.2 写 256。仅依据公开版本无法确定实际采用哪一个。"}

这条轨迹只有四步，但每一步都改变了 Agent 手里的信息。

第一步搜索的目标不是直接拿答案，而是定位一手资料。第二步把网页结果变成可按页检索的文档。第三步没有在命中第一个 “512” 后立刻停下，而是收集了全文的所有匹配项，于是撞见了附录里的 “256”。最后的回答保留冲突，没有擅自挑一个顺眼的数字。

中间状态应该保存什么

纯文本 Search 的状态并不等于“到目前为止的聊天记录”。更实用的做法是单独维护一个证据表。模型下一轮需要的是压缩后的事实，原文、页码和 URL 则留给引用与回放。

evidence_store.json
{
  "question": "DeepMMSearch-R1 的 RL batch size 是多少？",
  "known": [
    {
      "claim": "正文写 batch size = 512",
      "source": "paper_0#page=5",
      "status": "supported"
    },
    {
      "claim": "附录写 batch size = 256",
      "source": "paper_0#page=18",
      "status": "supported"
    }
  ],
  "open_questions": ["哪个数字对应最终实验？"],
  "answer_policy": "报告冲突，不自行推断"
}

这里也能看出 query 是怎样变化的。第一条 query 面向搜索引擎，用于找文档；第二条 query 面向文档内部，用于找字段。若继续追查，还可以搜作者仓库里的配置文件。不同工具吃的 query 并不一样，简单地把用户原话反复塞进搜索框，通常只会得到一堆重复页面。

什么时候可以停

停止条件不是“工具次数用完了”，而是答案中的重要断言已经有出处，剩余的不确定性也被明确写出。这个例子没有彻底消除疑问，但继续浏览同一份 PDF 不会产生新证据。此时诚实地报告版本内冲突，比编一个确定答案更好。

二、多模态 Search：从一张照片查到实体

多模态 Search 多了一件麻烦事：问题的关键线索可能不在文字里，而在图片的某个局部。为了看清区别，下面用一张桥梁照片举例。用户问：“图中是哪座桥，哪一年开放？”

原图能看到斜拉桥轮廓、岸线和一块很小的路牌，但没有可直接复制的实体名。比较稳妥的顺序是先定位视觉线索，再确认实体，最后用文本来源核实年份。

multimodal_search_trace.jsonl
{"step": 0, "event": "question",
 "images": [{"image_id": "img_0", "path": "bridge.jpg"}],
 "text": "图中是哪座桥，哪一年开放？"}

{"step": 1, "event": "tool_call", "tool": "crop",
 "args": {"image_id": "img_0", "box": [418, 86, 982, 536]},
 "decision": "桥塔和岸线比整张照片的天空、水面更有辨识度"}

{"step": 1, "event": "tool_result",
 "image_id": "img_1", "parent_id": "img_0", "artifact": "crop_001.jpg"}

{"step": 2, "event": "tool_call", "tool": "image_search",
 "args": {"image_id": "img_1", "top_k": 5}}

{"step": 2, "event": "tool_result",
 "candidates": [
   {"label": "Kessock Bridge", "score": 0.87},
   {"label": "Skye Bridge", "score": 0.41}
 ]}

{"step": 3, "event": "tool_call", "tool": "search_web",
 "args": {"query": "Kessock Bridge opened official year"},
 "decision": "图搜只确认候选实体，开放年份另找文本证据"}

{"step": 3, "event": "tool_result",
 "items": [{"title": "Kessock Bridge", "snippet": "The bridge opened in 1982 ..."}]}

{"step": 4, "event": "final",
 "answer": "图中是 Kessock Bridge，1982 年开放。"}

多出来的是“视觉状态”

img_1 不是一张用完即丢的临时图。它是从 img_0 派生出来的新观察，后面的图片搜索明确引用它。只记录“调用过 crop”还不够，系统必须知道裁了哪张图、坐标是多少、输出去了哪里。

visual_state.txt
img_0  bridge.jpg
└── img_1  crop(box=[418, 86, 982, 536])
    └── image_search → Kessock Bridge (0.87)

text_search("Kessock Bridge opened official year")
└── evidence → opened in 1982

这棵很小的状态树解决了两个常见问题。第一，回放时能确认模型搜的是整图还是桥塔局部；第二，如果实体识别错了，可以回到 img_0 换一个区域，而不是从头重跑整条轨迹。OCR、超分辨率、透视矫正也是同样的道理：它们都在产生新的可引用视觉对象。

图搜结果不能包办事实核验

图片搜索返回的是相似候选，不是“这座桥 1982 年开放”的证据。它适合把未知视觉对象映射成一个可以继续检索的名字；开放年份、设计者、地点等属性，仍然要去文本来源里核对。多模态 Search 经常是图像负责找实体，文本负责查属性，而不是从头到尾只用一种工具。

纯文本与多模态，轨迹差在哪里

纯文本 Search多模态 Search
初始状态问题、已有文字问题、原图及 image_id
常用动作搜索、打开网页、文档内查找裁剪、OCR、图搜，再接文本搜索
观察结果网页、段落、表格新图片、视觉候选、文字证据
证据定位URL、页码、段落image_id、父图、操作参数、URL
常见失败摘要当原文、只看首个命中整图噪声大、派生图片丢失、相似图当事实
停止条件断言有文本出处实体和属性都得到独立核验

两者的循环没有本质区别，区别在状态空间。纯文本 Agent 主要管理文档和段落；多模态 Agent 还要管理一组不断派生的图片。如果 Harness 仍然只接受一个固定的 image 字段，模型即使学会了 crop，也很难在下一步准确引用刚裁出来的局部图。

把两种轨迹装进同一个 Harness

下面是一版刻意缩小的运行循环。工具可以返回文本，也可以注册一个新视觉对象；所有动作先写事件日志，再把精简后的 observation 送回模型。

search_harness.py
def run_search(question, images, model, tools, max_steps=12):
    state = SearchState(question=question)
    for image in images:
        state.visuals.add_root(image)

    for step in range(max_steps):
        action = model.next_action(state.model_view())

        if action.kind == "final":
            state.log.final(step, action.answer, state.evidence.used_ids())
            return action.answer

        if not tools.validate(action):
            state.observe(error="invalid tool arguments", retryable=True)
            continue

        result = tools.call(action.name, action.arguments)

        if result.image is not None:
            image_id = state.visuals.add(
                image=result.image,
                parent_id=action.arguments.get("image_id"),
                operation=action.name,
            )
            observation = {"ok": True, "image_id": image_id}
        else:
            observation = compact_text_result(result)

        state.log.tool(step, action, result.artifact, result.latency_ms)
        state.observe(observation)

    raise SearchBudgetExceeded(max_steps)

这里故意没有保存模型的隐藏思维文本。调试需要的是简短的 decision summary、工具参数、原始返回物、精简 observation、耗时和错误码。它们足够解释系统做了什么，也便于比较两次运行。若只保存最终对话，网页正文被截断、crop 坐标错位、图搜超时这些问题都会混成一句“模型答错了”。

一条可用轨迹的最低标准：动作能执行，观察能回填，证据能定位，派生图片能追溯，结束原因能解释。运行时轨迹和训练样本不是一回事；后者怎样筛选和用于 SFT / RL，我放在下一篇单独讨论。

结尾

纯文本 Search 的难点主要是找对文档、读完整证据、处理来源冲突；多模态 Search 在此基础上又加了一层视觉状态管理。把轨迹按事件摊开后，很多“Agent 能力问题”会变成很具体的工程问题：query 有没有改写，工具返回有没有丢字段，派生图片能不能被下一轮引用，答案是否真的覆盖了证据。

对我来说，能把这些问题从日志里回答清楚，才算真的跑通了一条 Agent Search 轨迹。
