多模态 DeepSearch 的训练路线

把论文放到同一张工程图里看：模型输出什么动作，环境提供什么工具，冷启动数据教会什么，强化学习又在优化什么。

01. 调研范围：只讨论会主动搜索的模型

这里把多模态 DeepSearch 限定为一类闭环系统：模型能看图或局部裁剪，主动选择文本搜索、图片搜索、网页访问等工具，并根据返回证据继续推理。固定的“先检索、再回答”流水线不在本文范围内。

读这批论文时，我主要记录四个问题：

  1. 模型输出怎样的 action，Harness 如何解析；
  2. 工具返回纯文本，还是包含新的图片 observation；
  3. SFT 数据从哪里来，哪些 token 计算 loss；
  4. RL 的奖励、工具成本和失败轨迹怎样处理。
日期说明：这是一份截至 2026 年 7 月的工作笔记。对“开源”的判断以当时公开的论文、仓库、数据与训练脚本为准，不把论文中的 release promise 当成已经发布。

02. 代表工作放在一张图里

这条路线的变化并不是标签越来越复杂，而是工具空间、搜索长度和训练数据逐步接近真实环境。

MMSearch-R1 的公开仓库把图片搜索实现为 SerpAPI，把文本搜索组合为 SerpAPI、Jina Reader 与摘要模型。这个细节很重要：论文里的“文本搜索”通常不是一个 API，而是一条检索、抓取、清洗、压缩的子流水线。

03. 工具协议：重要的是可解析，不是长得像谁

论文中可以看到专属标签、XML 包 JSON，以及原生 function calling。它们都在解决同一个问题：把自由文本中的“意图”收敛成 Harness 可以执行的动作。

tool_protocol.txt · One action per turn
<think>
图片左下角的徽标可能是实体识别线索，先裁剪再做反向图片搜索。
</think>
<tool_call>
{
  "name": "image_search",
  "arguments": {
    "image_id": "img_0",
    "bbox": [0.08, 0.61, 0.34, 0.93],
    "top_k": 5
  }
}
</tool_call>

协议设计最好保持三个不变量：

  • 一回合只产生一个 action，降低解析歧义；
  • 图像坐标统一约定为像素或归一化坐标，不能混用；
  • 训练时的终止方式与推理时一致，包括 EOS 和 stop string。
validate_bbox.py · Python
def validate_bbox(box):
    if not isinstance(box, list) or len(box) != 4:
        raise ValueError("bbox must be [x1, y1, x2, y2]")

    x1, y1, x2, y2 = map(float, box)
    if not all(0.0 <= value <= 1.0 for value in box):
        raise ValueError("bbox must use normalized coordinates")
    if x1 >= x2 or y1 >= y2:
        raise ValueError("bbox has non-positive area")

    area = (x2 - x1) * (y2 - y1)
    if area < 0.0025:
        raise ValueError("crop is too small to search reliably")
    return [x1, y1, x2, y2]

04. 冷启动 SFT：不是教答案，而是教轨迹

DeepMMSearch-R1、Vision-DeepResearch 和 OpenSearch-VL 都把冷启动放在 RL 之前。原因很朴素：复杂工具空间里，基础模型未必能稳定产生合法 action，更谈不上在长轨迹中利用 observation。

一条训练样本最好同时保留“模型看到的消息”和“环境侧审计信息”。前者进入 tokenizer，后者用于重放与分析。

trajectory.json · Training record
{
  "sample_id": "vqa_01842",
  "messages": [
    {"role": "user", "content": ["question", "img_0"]},
    {"role": "assistant", "content": "<think>...</think><tool_call>..."},
    {"role": "tool", "content": "results: [...]"},
    {"role": "assistant", "content": "<think>...</think><answer>..."}
  ],
  "audit": {
    "raw_tool_responses": ["s3://bucket/run/01842/step_0.json"],
    "tool_latency_ms": [842],
    "tool_errors": [],
    "answer_source_ids": [2, 4]
  }
}

构造数据时最危险的捷径，是问题文本已经泄露图中实体，或者整图反向搜索能一步命中。OpenSearch-VL 使用 Wikipedia 路径采样、模糊实体改写和 source-anchor visual grounding，目标正是减少这类“一步检索坍缩”。

Loss mask：通常对 assistant 的 think、tool call 与 answer 计算损失，对 user 和 tool observation 屏蔽损失。模型要学的是怎样使用 observation，而不是复述搜索引擎返回值。

05. Agentic RL 优化的是决策顺序

当格式已经稳定，RL 才开始优化策略：该不该搜、先搜图还是先搜文本、什么时候访问网页、证据是否足够，以及是否值得再花一次工具预算。

rollout.py · Pseudocode
for turn in range(max_turns):
    output = policy.generate(messages)
    action = parse(output)

    if action.is_answer:
        break

    result = environment.execute(action)
    messages += [assistant(output), tool(result.observation)]
    trace.append({"action": action, "result": result})

reward = answer_reward(output, gold)
reward += evidence_reward(trace, gold_sources)
reward += format_reward(trace)
reward -= tool_cost(trace)
reward -= repeated_query_penalty(trace)

只用最终答案奖励会留下两个漏洞：模型可能靠参数记忆答对；也可能用大量无效搜索偶然命中。证据覆盖和工具成本不是装饰项，它们决定训练出来的是“会搜索”，还是“会挥霍 API”。

06. 长轨迹里，工具失败会污染后半程

在线环境的失败具有级联效应。一次网页访问超时，后续所有 action 可能都建立在缺失证据上。OpenSearch-VL 的 fatal-aware GRPO 正面处理了这个问题：失败后的 token 不再作为有效学习信号，同时尽量保留失败前的有用推理。

failure_mask.py · Python-like
def build_failure_mask(turns):
    mask = []
    fatal_seen = False

    for turn in turns:
        if turn.tool_result.is_fatal:
            fatal_seen = True

        # 失败发生前的模型 token 仍可参与更新；
        # 失败后的级联轨迹不再提供可靠优势估计。
        mask.extend([0 if fatal_seen else 1] * turn.num_policy_tokens)

    return mask

实现时还应区分 retryable 和 fatal。超时、限流可能值得重试；参数越界、图片不存在或权限拒绝，继续同一个动作通常只会浪费预算。

07. 评测中最容易被忽略的三类泄漏

  1. 参数记忆泄漏：模型不搜索也能直接回答，最终准确率无法证明搜索有效。
  2. 文本线索泄漏：问题已经写出视觉实体，视觉检索退化成普通文本搜索。
  3. 近重复图片泄漏：整图反向搜索直接命中原网页，局部理解和多跳推理都被绕过。

VDR-Bench 把这些问题放到评测中心：强调 visual-search-centric 问题、真实噪声、多轮裁剪搜索，并包含 2,000 个经多阶段构造和专家审核的 VQA 样本。MMSearch-Plus 则进一步强调来源感知、细粒度视觉线索传播与检索噪声下的交叉验证。

评测建议：至少同时报告 direct-answer baseline、工具调用成功率、平均调用次数、证据命中率和最终答案分数。缺少 direct-answer baseline，就无法判断模型是否真的使用了搜索。

08. 一条更稳妥的复现路线

  1. 先用固定缓存实现文本搜索，跑通多轮 message loop；
  2. 加入局部裁剪和图片搜索，但把图像 observation 固定到本地对象存储；
  3. 用少量人工检查的轨迹做 SFT，只验证格式、工具选择和停止条件；
  4. 建立 direct-answer 与 search-enabled 两套评测；
  5. 最后接在线 API 做 RL，并记录每次外部响应以便重放。

不要一开始就把真实搜索、长轨迹、SFT 和 RL 同时接上。那样任何波动都很难归因。先让环境可重放，再讨论奖励。

这批工作的共同结论：多模态 DeepSearch 的门槛已经从“模型会不会输出搜索标签”，转移到训练数据是否避免捷径、工具环境是否能承受失败、以及评测能否证明证据真的来自搜索。
