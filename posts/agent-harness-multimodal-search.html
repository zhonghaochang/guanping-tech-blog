多模态 DeepSearch 的训练路线：四篇工作的数据、工具与 Harness

只看 benchmark 分数，很难判断一个系统到底学会了搜索，还是碰巧接上了更强的搜索服务。本文把四篇工作放进同一套坐标：数据怎样生成、模型看见什么 observation、工具如何执行、SFT/RL 怎样训练，以及长轨迹失败后如何处理。

常中豪2026 年 7 月 12 日

调研对象是 MMSearch-R1、DeepMMSearch-R1、Vision-DeepResearch 和 OpenSearch-VL。它们大致构成一条清楚的演进线：从两种搜索工具、最多几轮的短轨迹，走向七种工具、64K 上下文和 7 万 token response budget 的长程研究环境。

口径说明：本文数字以论文正文、附录与官方仓库为准。原始问题数、生成轨迹数、过滤后轨迹数和最终训练样本不是同一个口径，我会分别标出。论文没有披露的配置不做推测。

1. 先看结论：差异主要不在模型，而在环境

工作基础模型SFTRL工具规模典型轨迹上限
MMSearch-R1Qwen2.5-VL-7B-Instruct主方案无；另做 8K 对照约 5K FVQA图片 + 文本搜索正文称 3 轮、最多 2 次搜索
DeepMMSearch-R1Qwen2.5-VL-7B-Instruct10K约 5K FVQA全文/裁剪图搜 + 文搜最多 10 次工具调用
Vision-DeepResearchQwen3-VL-8B / 30B-A3B30K15K视觉搜索 + 网页 + Python50 轮、64K context
OpenSearch-VLQwen3-VL-8B / 30B-A3B / 32B36,5928K7 种工具70K response tokens

真正决定难度的，是 action space 和 observation space。只有文本搜索时，模型主要学习 query 改写；加入图片搜索和裁剪后，它还要选择视觉区域；再加入超分、透视矫正和 OCR，模型必须判断“当前失败是实体未知，还是像素质量不够”。工具越多，SFT 冷启动越重要，Harness 的状态管理也越难。

2. MMSearch-R1：先证明“不做 SFT 也能学会搜”

MMSearch-R1 的核心实验很克制：Qwen2.5-VL-7B-Instruct 在简单的图片搜索和文本搜索环境中直接做 GRPO，不依赖冷启动 SFT。它想回答的不是“能否建一个最强系统”，而是“纯 RL 能否从答案奖励里涌现主动搜索行为”。

数据是怎样凑成约 5K 的

  • 先从 10K visual concepts 构造 6,000 条自动视觉概念数据，其中 5,400 条训练、600 条测试。
  • 加入 7,000 条 InfoSeek textual-knowledge 样本，以及 800 条人工数据。
  • 经过搜索需求平衡后，FVQA-train 约 5,000 条：3,400 条必须搜索，1,600 条无需搜索。
  • 测试集 1,800 条，来自三个来源，每个来源 600 条。

这里的 10K、6K、7K 和 5K 属于不同阶段。论文最终用于 RL 的并不是它们简单相加，而是经过任务构造和 search balancing 的 FVQA 训练集。

工具和 Harness

官方仓库的图片搜索用 SerpAPI，返回前 5 个图片匹配及其页面；文本搜索同样从 SerpAPI 取前 5 个结果，再用 Jina Reader 读取网页，最后由 Qwen3-32B 生成聚焦查询的摘要。协议采用专用标签：

MMSearch-R1 · action protocol
<reason>需要先确认图中实体。</reason>
<search><img></search>

<information>图片候选与对应页面……</information>

<reason>实体已确定，继续核验属性。</reason>
<text_search>entity attribute</text_search>

<answer>最终答案</answer>

正文写的是每条 rollout 最多 3 个对话轮次、最多 2 次搜索，第三轮给答案，而且图片搜索只能在第一轮使用。附录 F.1 又写“每条 rollout 最多 3 次工具调用”。这是论文内部的表述差异；实现复现时应以代码中的环境上限为准，不能只摘一处文字。

RL 与 SFT 对照配置

  • RL：veRL + GRPO；每 step batch 512，每题 8 个 rollouts。
  • 硬件：4 个节点、每节点 8 张 H100，共 32 张 GPU。
  • 学习率 2e-6，KL β=0.001，clip=0.2；论文报告约在第 50 step 收敛。
  • reward：字符串精确匹配为主；答对但使用搜索会乘以 0.1 的惩罚因子；格式项权重 α=0.1。

为了比较 SFT，作者还让 GPT-4o 生成 13.2K 条轨迹，过滤掉 5.2K 条答案错误轨迹，得到 8K。Qwen2.5-VL-7B 用 LLaMA-Factory 在 8 张 H100 上训练 1 epoch，学习率 1e-5，cosine schedule，10% warmup；prompt 和工具返回被 mask。

我的判断：MMSearch-R1 证明的是“小 action space 下 RL 可以从零学会是否搜索”。当工具增加到裁剪、OCR、网页访问和图像增强时，纯 RL 的探索空间会快速膨胀，这个结论不能直接外推。

3. DeepMMSearch-R1：用 10K SFT 轨迹解决多工具冷启动

DeepMMSearch-R1 仍以 Qwen2.5-VL-7B-Instruct 为基础，但把“识别哪里需要搜”和“怎样进行多步图文搜索”先写进 SFT，再用 RL 优化。它的数据漏斗比最终的 10K 更值得关注。

从 20 万问题到 1 万训练轨迹

  1. 从 InfoSeek 随机选择 200,000 个问题。
  2. 让 Gemini-2.5-Pro 在真实工具环境中生成多轮、带工具标签的轨迹。
  3. 只保留最终预测与 ground truth 一致的约 47,000 条。
  4. 按任务 taxonomy 平衡，得到 10,000 条 SFT 数据，并让 search-required 与 search-free 各占一半。

最终 10K 中，轮次数分布非常偏：1/2/3/4/5/6 轮分别为 5000、3289、1455、184、58、14。工具占比为文本搜索 68.9%、整图搜索 13.5%、裁剪图搜索 17.6%。换句话说，长轨迹很少，模型首先学到的是“少量步骤内正确选工具”。

图片搜索不只是把整张图发出去

它支持整图和 referring expression 两种入口。若模型指定局部实体，Harness 先用 GroundingDINO 得到 bounding box 并裁剪，再把局部图送到内部图片搜索；前 5 个结果由 GPT-5-chat-latest 汇总。文本搜索从内部 web index 取前 5 个结果，也由 GPT-5-chat-latest 汇总。

DeepMMSearch-R1 · 三类动作
<img_search>whole image</img_search>

<img_search>the red emblem on the upper-left corner</img_search>

<text_search>identified entity founding date</text_search>

环境限制为最多一次整图或裁剪图搜索，可以做多次文本搜索，总工具调用不超过 10 次。这个限制本身就是策略先验：视觉搜索负责锁定实体，文本搜索负责核验知识。

4. Vision-DeepResearch：把轨迹拉到 50 轮和 64K

Vision-DeepResearch 使用 Qwen3-VL-8B-Instruct 与 Qwen3-VL-30B-A3B-Instruct，目标从“找一个实体”进一步扩展到长程视觉研究。它的突出点有两个：把视觉搜索轨迹桥接到成熟的文本 DeepResearch 模型，以及用异步调度解决真实工具 rollout 的吞吐问题。

30K SFT + 15K RL

  • SFT 共 30K：16K verified factual VQA、8K text-only QA、6K fuzzy VQA。
  • RL 共 15K 高质量 VQA：10K 过滤后的现有数据 + 5K fuzzy VQA。
  • RL 使用真实在线视觉搜索、文本搜索和网页访问，最长 50 轮，context 64K，每轮 response 最多 4K。
  • reward 是纯准确率信号，由 LLM judge 输出 1 或 0。

数据生成先让多模态模型进行多实体、多尺度 crop/search：图片搜索后访问网站，再由辅助 MLLM 汇总网页；外部 judge 判断是否命中。随后把图片转成详细文字描述，将已经定位的实体和证据交给文本 DeepResearch foundation model，继续网页搜索、访问、总结或运行 Python。最后用 rejection sampling 验证答案。

Vision-DeepResearch · 数据桥接的抽象
visual_prefix = multimodal_agent.run(
    image=image,
    tools=[crop, image_search, website_visit, webpage_summary],
)

text_state = {
    "question": question,
    "image_description": describe(image),
    "identified_entities": visual_prefix.entities,
    "evidence": visual_prefix.evidence,
}

full_trace = text_deepresearch_agent.continue_run(
    state=text_state,
    tools=[web_search, website_visit, webpage_summary, python],
)

keep = answer_judge(full_trace.answer, ground_truth)

异步 Harness 与异常轨迹

同步 rollout 会让所有 GPU 等最慢的网页。论文采用类似 rLLM 的队列调度器和工具池，让模型生成、工具调用和 observation 回注异步推进，并允许多个 crop 请求并行；报告吞吐超过同步方案 10 倍。

稳定性处理也更像生产系统：n-gram 重复检测；连续 3 次格式或工具错误后终止；单次错误则返回明确 observation 允许重试。超过预算、重复或错误步数超过一半的异常轨迹，不参与反向传播，但仍保留在 advantage 计算中。64K 上下文下 FP16 容易 overflow，因此使用 BF16。

没有披露的部分：论文 PDF 没有给出完整的 GPU 数量、学习率等训练配置。能够确认的是数据规模、50 轮、64K context、4K 单轮 response、纯准确率 reward 和异步 rollout 设计；其余不应补写。

工具组合的消融很直观：Direct 平均 12.0；整图搜索 WIS 16.0；WIS + 文搜 29.3；裁剪图搜 CIS 23.0；CIS + 文搜达到 40.0。这说明视觉定位和文本核验不是替代关系，而是两段互补过程。

5. OpenSearch-VL：七种工具与可追踪的视觉状态

OpenSearch-VL 把多模态 DeepSearch 做成了更完整的开放环境。基础模型包括 Qwen3-VL-8B-Instruct、30B-A3B-Instruct 和 32B-Instruct，工具不只负责找信息，也负责改善“可看见性”。

工具后端实现解决的问题
TextSearchSerper top-k → Jina Reader → Qwen3-32B 生成每篇 2–4 句摘要查实体属性和网页证据
ImageSearchPolaris Lens API由公开图片 URL 找相似图片、实体、页面与 caption
SharpenOpenCV unsharp mask增强轻微模糊边缘
SuperResolutionOpenCV dnn_superres + EDSR，失败回退原图放大低分辨率局部
PerspectiveCorrectCanny → 最大四边形 → 四点变换校正文档、招牌透视
Crop确定性裁剪并保存到磁盘创建可复用的局部视觉状态
OCR远程 PaddleX layout-aware parsing读取复杂版式文字

Harness 的关键不是 schema，而是 active visual context

工具用 OpenAI 风格 function schema 注入 <tools>，每轮恰好输出一个 <tool_call> JSON。特别重要的是视觉上下文单调增长：原图、裁剪图、超分图和透视矫正图都保留引用，后面的工具可以继续操作某张派生图，而不是每轮回到原图。

visual_state.json · 派生图片可继续被引用
{
  "active_visual_context": [
    {"image_id": "img_0", "path": "question.jpg", "parent": null},
    {"image_id": "img_1", "path": "crop_001.jpg", "parent": "img_0",
     "operation": {"name": "Crop", "box": [420, 90, 980, 530]}},
    {"image_id": "img_2", "path": "sr_001.png", "parent": "img_1",
     "operation": {"name": "SuperResolution", "scale": 4}}
  ],
  "next_action": {
    "name": "OCR",
    "arguments": {"image_id": "img_2"}
  }
}

文本和 OCR observation 只作条件并被 loss mask；图片 observation 同样不承担 token loss，但会通过视觉编码器进入后续上下文。这正是多模态 Harness 和纯文本 Agent 最大的结构差异。

36,592 条专家轨迹是怎样来的

  • 从 Wikipedia 构造长度 h∈{2,3,4} 的实体路径，并进行 fuzzy entity rewriting 和 source-anchor grounding。
  • 合并 LiveVQA、FVQA 与 WebQA；用 Qwen3-VL-32B 过滤无需工具即可回答，以及只做一次 ImageSearch 就能回答的简单题。
  • 随机退化 10% 图片，迫使轨迹使用图像增强工具。
  • Claude Opus 4.6 在真实工具环境中每题独立采样 K=5 条轨迹。
  • 先由 GPT-4o 判断答案，再由 GPT-5.4 判断过程和工具使用，最终保留 36,592 条，平均 6.3 个工具轮次。

SFT 与 RL 的工程规模

阶段数据与序列训练配置
SFT36,592；cutoff 32K；8 epochs全参数；vision tower/projector 均不冻结；ZeRO-3；bf16；256 GPU（32×8）；effective batch 256；lr 2e-5；warmup 0.1

RL 采用 RLOO within GRPO。每题采样数 8B 为 8、30B 为 16，temperature 0.7；8B 训练 100 epochs，30B 训练 5 epochs。Megatron-LM 侧使用 TP4、PP2，context parallel 为 8B 的 CP8 / 30B 的 CP4，MoE 还使用 EP8、ETP1。

reward 写成：

OpenSearch-VL · reward 结构
reward = r_format * (0.8 * r_accuracy + 0.2 * r_query)

# r_format: 确定性协议校验
# r_accuracy: GPT-4o 判断最终答案
# r_query: GPT-5.4 判断查询和工具过程

连续 3 次工具错误被判 fatal。fatal 之后的 token 被 mask；fatal prefix 的 advantage 用 max(advantage, 0) 做单边截断，避免模型因为环境尾部错误而负向学习此前仍合理的动作。

7. 训练路线真正发生了哪些变化

变化一：从“会不会搜”转向“先修图还是先搜”

MMSearch-R1 的动作空间很小，答案奖励足以诱发搜索。OpenSearch-VL 的动作空间包含增强、几何变换、OCR、图搜和文搜。此时错误工具选择会改变后续可见状态，仅靠最终答案做稀疏探索非常困难，所以需要专家轨迹先建立 action prior。

变化二：observation 从文本变成可累积的视觉状态

裁剪图、超分图不应只是一次性的 tool output，而要有稳定 ID、父子关系和磁盘生命周期。否则模型看到“img_2”时，Harness 无法知道它源自哪次操作，训练复盘也无法定位错误。

变化三：同步训练变成分布式在线系统

70K response、2048 个并行工具调用意味着瓶颈不再只是 GPU。DNS、网页读取、图片下载、OCR 服务、缓存命中和第三方 API 限流都会影响 rollout 分布。论文里的“64 张 GPU”并不能完整表示实验成本。

变化四：失败处理开始进入优化目标

短轨迹里失败可以整条丢弃；50 轮轨迹里这样做会浪费大量有效 prefix。于是出现 abnormal trajectory mask、fatal token mask、连续错误阈值和单边 advantage clamp。它们不是边角工程，而是训练算法的一部分。

8. 如果从零复现，我会选哪条路线

  1. 先做两工具闭环。文本搜索 + 图片搜索，单次动作，最多 4–6 步，先确认 parser、缓存、日志和评测稳定。
  2. 做 5K–10K 高质量 SFT。过滤无需搜索的题，tool observation 全部 mask；保留每步 audit。
  3. 再加局部视觉工具。先 Crop，再加 OCR；每个派生图使用稳定 image_id 和 parent。
  4. 最后做在线 RL。每题多 rollout，先用答案 + 格式 reward，再谨慎加入过程 judge 和工具成本。
  5. 把 Harness 指标纳入实验。除了准确率，还记录平均工具数、空结果率、超时率、fatal rate、上下文长度和每题真实成本。
我的判断：多模态 DeepSearch 的提升，不只是“模型推理更长”。它来自训练数据、搜索后端、视觉状态管理和 rollout 系统的共同扩展。论文对比时，必须把 Harness 当成模型能力的一部分来读。
