从 <search> 到真实 API：多模态搜索 Agent 如何停下来、调用工具并继续推理

结合 2025—2026 年代表性多模态搜索与 DeepResearch 项目，拆解模型 token、消息结束、工具解析、搜索 API、Observation 回注，以及 Prompt、SFT 和 RL 分别承担什么作用。

2026 年 7 月 12 日·约 25 分钟阅读·多模态 Agent 搜索
目录 1. 核心心智模型 2. 模型何时停 3. Harness 四层结构 4. 四类工具协议 5. 完整搜索轨迹 6. 十二个项目对比 7. Prompt / SFT / RL 8. SFT 数据格式 9. RL Rollout 10. 实现建议
一句话结论：模型本身通常不直接访问搜索引擎。它生成一个结构化动作；Agent Harness 解析动作、调用 API、整理结果，再把结果作为新消息送回模型，开启下一轮推理。

1. 核心心智模型：模型输出动作，环境执行动作

看到下面的输出时，很多人会误以为模型内部“自动联网”了：

<think>
需要先根据球衣、人物和背景识别战队。
</think>
<search><img></search>

实际上，这段内容首先只是模型生成的一串 token。真正使它产生搜索效果的是外部程序中的映射规则：

if parse_action(model_output) == "image_search":
    result = image_search_api(original_image)
    messages.append(tool_observation(result))
模型生成动作
解析器识别
工具执行 API
结果转为 Observation
下一轮模型推理

因此，工具能力由两部分共同构成：

  • 模型：决定什么时候调用什么工具，并生成参数；
  • Harness：解释动作、执行工具、控制预算、把结果回注上下文。
没有 Harness，即使模型完美输出 <search>，也只会得到一段普通文字，不会产生 HTTP 请求。

2. 生成到 </search> 时，会立刻暂停吗？

答案是：不一定。实际存在两种实现。

2.1 完整生成当前 assistant 回合，再解析

MMSearch-R1 的公开 rollout 更接近这种方式：推理引擎先生成完整回答，通常直到 <|im_end|> 或 EOS；随后 Harness 解码整段字符串，用正则检查输出末尾是否含有图片搜索或文本搜索标签，再执行工具。

<|im_start|>assistant
<think>需要先识别图片。</think>
<search><img></search>
<|im_end|>

这时流程是：

  1. 模型生成当前消息；
  2. 模型生成 <|im_end|> 或 EOS，当前调用结束；
  3. Harness 检查搜索标签；
  4. 执行图片搜索；
  5. 把结果包装成新的用户消息或工具消息；
  6. 再次调用模型。

2.2 检测到闭合标签后，提前停止解码

另一类系统把 </tool_call> 配置为停止字符串，或者在流式生成时实时检测闭合标签。一旦完整工具调用产生，就中断本轮解码并执行工具。

stop_strings = ["</tool_call>", "</answer>"]

if stream_buffer.endswith("</tool_call>"):
    abort_current_generation()
    execute_tool()

OpenSearch-VL 的训练系统明确支持工具调用的中断与后续轨迹恢复，并规定每轮仅输出一个 <tool_call> 或最终 <response>。

2.3 <|im_end|>、EOS 与 stop string 的区别

概念含义由谁定义/执行
<|im_end|>当前 ChatML 消息结束Tokenizer 与聊天模板定义；模型学习何时生成
EOS当前生成序列结束Tokenizer 提供 token ID，推理引擎看到后停止
stop string人为指定的文本停止条件Harness / 推理引擎配置

</search> 也不一定是一个 tokenizer token。它可能被切成 <、/、search、> 等多个 token。多数 Harness 是先解码成字符串，再进行正则或 JSON/XML 解析。

3. Agent Harness 的四层结构

3.1 模型与 Prompt 层

系统提示词声明工具名称、参数 Schema、输出格式、每轮最大动作数和最终答案格式。强指令模型可以凭零样本或少样本提示，临时遵循新的工具协议。

3.2 动作解析层

负责把模型输出转换成结构化动作：

Action(
    name="text_search",
    arguments={"query": ["NAVI first Major championship year"]}
)

解析失败时，Harness 还需要决定：让模型重试、返回格式错误、终止轨迹，还是自动修复 JSON。

3.3 工具执行层

这一层保存 SerpAPI、Serper、Jina Reader 等密钥，真正发起 HTTP 请求。API Key 不应出现在模型上下文中。

3.4 Observation 与状态管理层

搜索结果不会原封不动地无限加入上下文。Harness 通常会执行 Top-K 截断、去重、网页 Reader、摘要、图片下载、图片句柄注册、搜索预算统计和运行记忆压缩。

4. 2025—2026 年项目中的四类工具协议

4.1 工具专属标签

<search><img></search>
<text_search>NAVI first Major title year</text_search>

代表项目是 MMSearch-R1。优点是简单、适合直接 RL；缺点是每增加一个工具，就需要增加新的标签和解析规则。

4.2 统一的 XML + JSON 工具调用

<think>
需要查询该战队的首个 Major 冠军年份。
</think>
<tool_call>
{
  "name": "text_search",
  "arguments": {
    "query": ["Natus Vincere first Counter-Strike Major title"]
  }
}
</tool_call>

这是训练型多模态搜索 Agent 中最常见的形式。Vision-DeepResearch、OpenSearch-VL、VSearcher、SimpleSearch-VL 和 MM-DeepResearch 都采用了相近的统一调用思路。新增 crop、visit、OCR 或 super_resolution 时，只需扩充工具 Schema。

4.3 自定义多动作标签

<query>...</query>
<clip>...</clip>
<tool name="fetch_webpage_text">...</tool>
<code>...</code>
<done>最终答案</done>

InterLV-Search 的 Harness 使用这一设计，分别解析搜索、裁剪、网页读取、Python 和结束动作。

4.4 原生 Function Calling / MCP

模型 API 直接返回结构化的 tool_calls 字段，而不是在可见文本里输出 <search>。BrowseComp-V3 的 OmniSeeker 参考运行器使用 OpenAI 兼容函数调用，并通过 MCP 工具服务器执行 TextSearch、ImageSearch、ReverseImageSearch、WebVisit 和 CropImage。

{
  "role": "assistant",
  "content": null,
  "tool_calls": [{
    "id": "call_01",
    "function": {
      "name": "TextSearch",
      "arguments": "{\"query\":\"NAVI first Major title\"}"
    }
  }]
}

5. 一条模拟真实逻辑的完整搜索轨迹

下面使用统一 <tool_call> 协议，模拟一个图片识别加事实查询任务。格式与多个项目的公开 Prompt 一致，但具体问题为演示构造。

第 0 步：Harness 把工具 Schema 放入系统提示词

<tools>
{"name":"image_search","parameters":{"image":"string","bbox":"[x1,y1,x2,y2]"}}
{"name":"text_search","parameters":{"query":"array[string]"}}
{"name":"visit","parameters":{"url":"array[string]","goal":"string"}}
</tools>

规则：每轮只能输出一个 tool_call；工具调用后结束当前消息并等待结果。

第 1 轮：模型请求局部图片搜索

<think>
球衣胸前的徽标是识别战队的关键，应先裁剪并进行反向图片搜索。
</think>
<tool_call>
{
  "name": "image_search",
  "arguments": {
    "image": "img_1",
    "bbox": [410, 270, 590, 470]
  }
}
</tool_call>
<|im_end|>

推理引擎停止后，Harness 解析 JSON，裁剪图片、上传公开对象存储，再调用 Lens/反向图片搜索 API。

第 1 次外部 API 请求

POST /search
Authorization: Bearer ${SEARCH_API_KEY}

{
  "engine": "google_lens",
  "url": "https://image-host.example/crop_410_270_590_470.jpg",
  "num": 5
}

搜索服务返回原始 JSON，Harness 标准化为：

[
  {
    "rank": 1,
    "title": "Natus Vincere official logo",
    "thumbnail_url": "https://.../thumb.jpg",
    "image_url": "https://.../full.jpg",
    "source_url": "https://.../navi-team"
  }
]

如果框架支持视觉 Observation，它还会下载缩略图或高清图并注册为 img_2、img_3，让模型在下一轮真正看到候选图片，而不是只看到 URL。

第 2 轮：模型根据 Observation 改用文本搜索

<tool_response>
候选 1：Natus Vincere official logo
来源：https://example.com/navi-team
视觉匹配：高
</tool_response>

<think>
局部图片搜索已将战队定位为 Natus Vincere。现在需要查询其首个
Counter-Strike Major 冠军年份。
</think>
<tool_call>
{
  "name": "text_search",
  "arguments": {
    "query": [
      "Natus Vincere first Counter-Strike Major championship year",
      "NAVI first Major title history"
    ]
  }
}
</tool_call>

第 3 轮：模型决定访问来源网页

<tool_response>
1. PGL Major Stockholm 2021 — snippet: NAVI claimed their first Major...
2. Natus Vincere history — snippet: ...
</tool_response>

<think>
搜索摘要已给出候选年份，但应访问权威来源确认“首个 Major”的表述。
</think>
<tool_call>
{
  "name": "visit",
  "arguments": {
    "url": ["https://example.com/official-history"],
    "goal": "确认 Natus Vincere 获得首个 Counter-Strike Major 冠军的年份"
  }
}
</tool_call>

visit 背后可能是 Jina Reader、Playwright 或自建抓取器。完整网页不一定全部进入主模型上下文；Harness 可以先截断、切块检索或用辅助模型提取证据。

第 4 轮：最终回答

<think>
权威来源确认 Natus Vincere 在 PGL Major Stockholm 2021 获得队史首个
Counter-Strike Major 冠军。
</think>
<answer>2021 年。</answer>
关键点：所谓“工具调用后继续生成”，在逻辑上通常是一次新的模型调用。上一轮输出、工具 Observation 和新一轮 assistant 起始标记共同组成新的上下文。

DeepMMSearch-R1
2025		学习到的工具调用标签 SchemaGrounding、整图/局部图片搜索、文本搜索；结果回注后持续多轮 rollout冷启动 SFT → 在线 RL；SFT 先教会标签、工具选择与查询构造

7. 模型原生理解 <search> 吗？Prompt、SFT 与 RL 各自教什么？

7.1 模型不会天然把标签连接到 API

Tokenizer 可能认识 <tool_call> 或把它拆成若干 token；模型也可能因通用指令训练而具备 JSON、XML 和函数调用先验。但它不会天然知道你项目中的 image_search 对应 SerpAPI，或 visit 对应 Jina Reader。

7.2 Prompt：声明临时协议

Prompt 告诉模型有哪些工具、参数是什么、何时使用、每轮调用几个以及最终答案格式。AgentVista、InterLV-Search、BrowseComp-V3 与 MMSearch-Plus 等评测型 Harness 主要依赖这种方式驱动现成模型。

7.3 Few-shot：展示一条完整交互

Few-shot 让模型看到“提出动作 → 停止当前回合 → 接收工具结果 → 根据 Observation 继续”的完整样例，比只给 JSON Schema 更容易稳定遵循协议。

7.4 SFT：稳定格式与基本策略

SFT 把合法格式、工具选择、query 构造、结果利用和停止条件写入参数。DeepMMSearch-R1、DeepEyesV2、Vision-DeepResearch、OpenSearch-VL、VSearcher、SimpleSearch-VL 和 MM-DeepResearch 都采用了 SFT 后再 RL 的路线。

7.5 RL：优化何时搜、搜什么、何时停

RL 并不主要教 JSON 语法，而是优化工具使用策略：搜索是否必要、使用图片还是文本、如何改写 query、是否访问网页、是否重复搜索，以及如何在预算内得到正确答案。

MMSearch-R1 是重要例外：它在较简单的双工具环境中展示了直接 GRPO 的可行性；而 DeepEyesV2 等复杂工具环境发现，缺少冷启动 SFT 时工具调用会不稳定。二者并不矛盾，差别主要来自工具空间、基础模型先验、奖励设计和训练数据难度。

8. SFT 数据到底长什么样？

一条工具轨迹可以被序列化成多轮聊天：

USER:
问题 + 图片

ASSISTANT:              # 计算 loss
<think>需要先识别图片实体</think>
<tool_call>{"name":"image_search", ...}</tool_call>

TOOL / ENVIRONMENT:     # 通常不计算 loss
<tool_response>搜索结果……</tool_response>

ASSISTANT:              # 计算 loss
<think>实体已识别，继续文本搜索</think>
<tool_call>{"name":"text_search", ...}</tool_call>

TOOL / ENVIRONMENT:     # 不计算 loss
<tool_response>网页结果……</tool_response>

ASSISTANT:              # 计算 loss
<think>证据充分</think>
<answer>最终答案</answer>

训练掩码的核心原则是：

用户问题 token              loss_mask = 0
模型 think/tool_call token   loss_mask = 1
工具 observation token      loss_mask = 0
模型最终 answer token        loss_mask = 1

工具结果不是模型生成的动作，因此不应该让模型学习“背诵搜索引擎返回内容”；模型需要学习的是如何读取 Observation 后生成下一步动作。

9. RL Rollout：搜索环境如何进入强化学习

RL 时，模型需要真正与 Harness 多轮交互：

for turn in range(max_turns):
    output = policy.generate(messages)

    if valid_tool_call(output):
        observation = environment.execute(parse(output))
        messages += [assistant(output), tool(observation)]
        continue

    if valid_answer(output):
        break

reward = answer_score
reward += format_score
reward -= tool_cost
reward -= invalid_call_penalty
update_policy(reward)

常见奖励项包括：

  • 最终答案正确奖励；
  • 工具格式合法奖励；
  • 来源或证据支持奖励；
  • 搜索和网页访问成本惩罚；
  • 重复搜索、无效参数和超时惩罚；
  • 过程奖励或失败前有效步骤保留。

真实在线 API 会带来搜索排序漂移、网页失效、Reader 失败和延迟波动。MM-DeepResearch 使用离线搜索环境降低训练成本；OpenSearch-VL 则专门讨论了工具失败导致长轨迹后半段失效的问题。

10. 面向后续 SFT / RL 项目的 Harness 设计建议

  1. 采用统一 JSON 协议：<tool_call>{"name":...,"arguments":...}</tool_call>，不要为每个工具发明新外层标签。
  2. 每轮一个动作：需要并行搜索时允许在一个工具调用中批量传入多个 query 或多个 region，降低解析复杂度。
  3. 明确停止条件：可在 </tool_call> 处停止，或生成到消息 EOS 后解析，但训练和推理必须保持一致。
  4. Observation 不计监督损失：只训练模型产生的 think、tool_call 和 answer。
  5. 分开记录动作和环境：轨迹中保存原始 token、解析后的动作、API 原始响应、标准化 Observation 和错误状态。
  6. 设置硬预算：固定 max turns、各工具调用次数、Top-K、Reader 长度、上下文长度和超时。
  7. 同时维护在线与重放环境：在线环境衡量真实可用性，固定缓存重放环境用于公平训练和可复现评测。
  8. 把格式能力和搜索策略分开评测：合法工具调用不等于工具真的提供了答案关键证据。
最终理解：<search> 不是搜索能力本身，而是一种动作语言。Prompt 定义动作语言，SFT 让模型稳定说这种语言，RL 优化什么时候说、说什么；Harness 则把语言翻译成真实世界中的搜索、访问、裁剪和计算。
