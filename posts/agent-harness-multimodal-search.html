多模态 DeepSearch 的训练路线

最近把 MMSearch-R1、DeepMMSearch-R1、Vision-DeepResearch 和 OpenSearch-VL 串着看了一遍。这篇记录它们怎样处理数据、工具调用和长轨迹训练，也记下几处复现时容易踩的坑。

常中豪2026 年 7 月 12 日

多模态 DeepSearch 这条线发展得很快。只隔几个月，任务已经从“看图以后要不要搜一下”，走到了“让模型自己裁图、OCR、修复图片，再连续搜几十轮”。单看某一篇会觉得方案差异很大，按时间放在一起反而很顺：每篇都在补上一代系统暴露出来的短板。

我主要关心四件事：训练数据从哪里来，SFT 和 RL 各做了什么，模型能调用哪些工具，以及 Harness 怎样撑住这些轨迹。先用一张表把规模对齐，后面再逐篇展开。

MMSearch-R1：先把“该不该搜”学会

MMSearch-R1 的切入点很干净：有些问题看图就能答，有些问题必须访问外部知识。模型既要会搜索，也要避免见图就搜。作者从 10K 个视觉概念出发，自动构造 6,000 条 visual-concept 数据（5,400 训练、600 测试），另外混入 7,000 条 InfoSeek 文本知识样本和 800 条人工样本。

真正用于主要 RL 实验的 FVQA-train 约 5,000 条，其中 3,400 条需要搜索，1,600 条不需要搜索。这个比例值得留意。如果全是 search-required，最省事的策略就是每道题先调工具；加上 search-free 样本以后，模型才会为无意义调用付出代价。

一条很短的工具协议

它的环境只提供图片搜索和文本搜索。图片搜索走 SerpAPI；文本搜索也是 SerpAPI 找页面，再用 Jina Reader 抽正文，最后让 Qwen3-32B 把网页压成摘要。论文正文描述的轨迹最多 3 轮、至多 2 次搜索，而且图片搜索只允许在第一轮发生。这套限制看起来保守，但非常适合验证一个问题：不用冷启动 SFT，纯 RL 能不能学会选择工具。

mmsearch_action.json
{
  "action": "image_search",
  "query_image": "input_image"
}

{
  "action": "text_search",
  "query": "identified entity + missing attribute"
}

主实验用 GRPO：batch size 512，每条题采样 8 个 rollout，4×8 张 H100，学习率 2e-6。作者也做了一条 SFT 对照线：让 GPT-4o 生成 13.2K 条轨迹，过滤掉 5.2K 条答案错误的样本，剩下 8K。结果里最有意思的不是“SFT 还是 RL 谁赢”，而是动作空间还很小时，RL 确实可以直接把搜索时机学出来。

DeepMMSearch-R1：局部图搜索让 SFT 回来了

到了 DeepMMSearch-R1，工具从两种变成三种：文本搜索、整图搜索和局部图搜索。局部图搜索并不是传一组裸坐标，模型先给 referring expression，Harness 用 GroundingDINO 找区域、裁图，再去内部图片索引检索。图片搜索和文本搜索的 Top-5 结果最后都由 GPT-5 压成模型可读的摘要。

动作一复杂，随机探索就贵了。作者先从 InfoSeek 随机抽 200,000 条题，用 Gemini-2.5-Pro 生成工具轨迹，只保留最终答案与 ground truth 一致的约 47,000 条，再做类型平衡，落到 10,000 条 SFT 数据；search-required 和 search-free 各占一半。

这些轨迹里，文本搜索占 68.9%，整图搜索占 13.5%，局部图搜索占 17.6%。也就是说，加入视觉工具以后，文本检索仍然是主力；视觉工具更多是在搜索前半段解决“图里这个东西叫什么”。

SFT 用 LoRA rank 8，只训练语言模型部分，视觉编码器和 projector 冻结；3 个 epoch，学习率 1e-4，1×8 张 H100，global batch 8。随后再做 RL，每条题最多一次图片 / 局部图搜索，可以多次文本搜索，总工具调用不超过 10 次。

复现时要注意一处原文冲突：RL 正文写 30 epochs、batch size 512，附录 B.2 写 20 epochs、batch size 256；两处都写了 4×8 H100、每题 8 个 rollout。公开版本本身无法判断哪组是最终配置，最好直接查看作者代码或向作者确认。

Vision-DeepResearch：50 轮以后，Harness 变成主角

短轨迹可以同步跑：模型生成一次，等工具返回，再继续生成。上限拉到 50 轮以后，这样做 GPU 会频繁等网络，搜索服务也会被一波一波的请求打满。Vision-DeepResearch 让我印象最深的地方不只是数据规模，而是它把训练吞吐当成核心问题处理。

它用 Qwen3-VL-8B 和 30B-A3B。SFT 共 30K：16K 可验证事实型 VQA、8K 纯文本 QA、6K fuzzy VQA；RL 共 15K：10K 过滤后的现有数据和 5K fuzzy 数据。上下文长度 64K，每轮最多生成 4K，整条轨迹允许 50 轮。

论文实现了异步任务队列和工具池。某些 rollout 在等搜索时，推理引擎可以继续处理别的轨迹，报告的吞吐比同步实现高 10 倍以上。这个提升来自调度，不是模型突然少思考了。

async_rollout.py
async def rollout(sample, policy, tool_pool):
    state = init_state(sample)

    while state.turn < 50:
        action = await policy.generate(state.context)
        if action.is_final:
            return finish(state, action)

        # 当前轨迹等待 I/O 时，调度器可以运行其他 sample
        result = await tool_pool.execute(action.tool, action.args)
        state.append(action, result)

    return truncate(state)

长轨迹还要认真定义失败。Vision-DeepResearch 会检测 n-gram 重复并提前结束；单次工具错误允许恢复，连续 3 次格式或工具错误才终止。异常轨迹不参与反向传播，但仍保留在 advantage 计算里。论文给出了准确率 judge 奖励，不过没有公开完整的 GPU 和学习率配置，这部分没必要从常见 recipe 里猜。

OpenSearch-VL：不只搜索图片，还会处理图片

OpenSearch-VL 把动作空间扩到 7 个工具：TextSearch、ImageSearch、Crop、OCR、Sharpen、SuperResolution 和 PerspectiveCorrect。这里的变化挺实用。模型遇到模糊路牌时，不必在“看不清”和“直接图搜”之间二选一，它可以先裁剪、超分、矫正，再把新图送去 OCR 或图片搜索。

工具背后的实现也写得比较具体：文本搜索用 Serper 找 Top-k 页面、Jina Reader 抽正文、Qwen3-32B 压成 2–4 句摘要；图片搜索用 Polaris Lens；OCR 用 PaddleX；超分辨率用 EDSR；其余视觉操作主要由 OpenCV 完成。

Active visual context 是关键细节

每次 Crop 或 SuperResolution 都会生成新图。如果只把新图塞回对话、不记录来源，几轮以后模型和日志都分不清自己正在操作哪一张。OpenSearch-VL 给图片分配 ID，并保留父子关系：

visual_context.json
{
  "img_0": {"source": "user_upload"},
  "img_1": {
    "source": "crop",
    "parent": "img_0",
    "box": [418, 86, 982, 536]
  },
  "img_2": {
    "source": "super_resolution",
    "parent": "img_1",
    "scale": 4
  }
}

数据生成同样更重。作者从 Wikipedia 实体路径以及 LiveVQA、FVQA、WebQA 等数据构题，过滤掉不需要工具和只靠一次整图搜索就能解决的样本，并主动劣化 10% 的图片。Claude Opus 4.6 每题采样 5 条专家轨迹，再由 GPT-4o 判断最终答案、GPT-5.4 判断过程，得到 36,592 条 SFT 轨迹，平均 6.3 个工具轮次。

SFT 这次是全参数训练，没有冻结 vision tower 和 projector：32K 截断、8 epochs、学习率 2e-5、有效 batch 256，论文写使用 256 张 H20。RL 再用一份不重叠的 8K VQA 数据，64 张 H20，最大 response 长度 70K。8B 模型每题 8 个 rollout，30B 模型每题 16 个；运行侧用 SGLang 异步调度，最多 256 个并行任务、2,048 个并行工具调用。

它对坏轨迹的处理也更细：连续 3 次工具错误视为 fatal，fatal 之后的 token 不算 policy loss，并对对应 advantage 做单边截断，避免环境故障反过来教坏策略。多工具 RL 如果不处理这类噪声，reward 再精致也容易被接口错误污染。

四篇连起来，路线就比较清楚了

MMSearch-R1 先证明，小动作空间里可以直接用 RL 学“何时搜索”；DeepMMSearch-R1 加入局部视觉动作，用 SFT 给模型一条可靠的起跑线；Vision-DeepResearch 把轨迹拉长，训练瓶颈转到异步调度和错误恢复；OpenSearch-VL 再把裁剪、OCR、超分、透视矫正都纳入策略，开始接近真正的主动视觉研究 Agent。

这不是四套互斥方案，更像四层逐步加上的能力：

    ↓

如果现在做一个最小复现

我不会一开始就照搬 7 个工具和 50 轮上下文。先做三种动作已经足够：文本搜索、整图搜索、Crop 后的局部图搜索。数据里同时保留需要搜索和不需要搜索的题，先用 5K–10K 条经过答案验证的轨迹做 SFT，再决定是否值得接 RL。

在训练之前，最好先把 Harness 的几件小事做扎实：每张派生图片有唯一 ID；工具返回既保留原始 artifact，也提供短 observation；超时和格式错有明确错误码；一条轨迹可以从事件日志完整回放。这些工作不出现在模型结构图里，却直接决定坏样本能不能被找出来。

最后才是扩大 rollout。先统计成功轨迹的工具轮数、重复 query、无效 crop 和超时率。如果多数题三步内就结束，贸然把上限提到 50 只会让模型学会绕路。长轨迹应该由任务需要推动，而不是拿最大轮数当能力指标。
