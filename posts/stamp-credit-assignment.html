STAMP 详解：如何把 DeepSearch 的最终奖励追溯到真正找到证据的搜索动作

从奖励—信用错配、训练期证据图、LLM 证据验证和 First Exposure 归因，一直讲到 GRPO 的步骤级优势调制与参考实现。

版本：arXiv:2607.11172v1，2026 年 7 月 13 日
本文定位：面向研究者的中文方法拆解、公式推导、实例演算与参考实现。

0. 一句话理解

传统 GRPO 会把整条轨迹的同一个优势值广播给所有模型动作；STAMP 则先验证最终引用的网页支持了哪些目标实体和关系，再沿 URL 的轨迹来源回溯到这些网页第一次被暴露的 search/open 步骤，最后只在优势层面对这些步骤进行更强的正向强化或更弱的负向惩罚。

核心链路是：

text
最终答案中的引用
      ↓
被 LLM 验证为支持目标实体/关系的文档
      ↓
轨迹日志中的 URL 首次暴露步骤
      ↓
步骤信用 c_t
      ↓
步骤级优势 A*_{i,t}
      ↓
该 action step 内模型生成 token 的策略梯度

它解决的不是“如何给整条轨迹打更好的分”，而是：

一条成功或失败的长搜索轨迹中，究竟哪些动作应承担更多信用？

1. 为什么 DeepSearch 特别需要信用分配

DeepSearch/DeepResearch 的一条轨迹通常包括：

  • 多轮问题拆解；
  • 多次 search；
  • 多次 open/read；
  • 实体消歧；
  • 证据交叉验证；
  • 最终答案和引用生成。

普通结果监督只得到一个终局奖励：

\[ R(\tau) \in \{0,1\} \]

假设一条轨迹有 12 个动作，其中只有第 3、8 步真正找到关键证据，但最终答对了。普通 GRPO 会让 12 个动作共享同一个正优势。相反，如果最终答错，即使第 3 步曾经找到正确证据，也会和错误动作一起受到负向更新。

论文将这种现象称为：

奖励—信用错配（reward-credit mismatch）。

论文统计显示：

  • 正确轨迹中有 83.5% 的步骤没有产生目标证据；
  • 错误轨迹中仍有 7.0% 的步骤暴露了有效实体或关系证据；
  • 无目标证据轨迹准确率为 18.0%；
  • 只有实体证据时为 30.8%；
  • 获得经过关系验证的证据时为 69.1%。

这些结果说明：最终结果和局部动作价值并不完全一致。

2. 先区分四个层次：Reward、Credit、Advantage、Gradient

层次 作用 STAMP 是否改变
轨迹奖励 \(R_i\) 判断整条 rollout 的最终质量 否
组相对优势 \(A_i\) 将轨迹奖励转成相对训练信号 先按普通 GRPO 计算
步骤信用 \(c_{i,t}\) 判断某一步是否首次暴露有效证据 是，STAMP 新增
Token 优势 \(A^*_{i,m}\) 决定模型生成 token 的更新强度 是，按步骤调制

因此，STAMP 并不是一个新的终局奖励函数，而是一个优势重分配机制。

3. 训练期证据图：目标实体与目标关系是什么

每个训练问题都配有一张训练期证据图：

\[ G=(V,E) \]

其中：

  • \(V\)：问题涉及的实体节点；
  • \(V_{\text{tar}}\subseteq V\)：需要通过搜索发现的隐藏目标实体；
  • \(E\)：实体之间的事实关系；
  • \(U_{\text{tar}}\)：用于验证的原子关系单元。

3.1 具体问题

考虑问题：

哪位演员出演过电影 Sergeant York，并曾与一位出演电视剧 Bewitched 的女演员结婚？

训练期标准答案和中间实体为：

  • \(E_0=\) Gig Young：最终目标演员；
  • \(E_1=\) Elizabeth Montgomery：中间隐藏实体。

问题中直接出现的：

  • Sergeant York；
  • Bewitched；

属于已知锚点实体，不产生实体发现信用。

目标关系可以拆成：

\[ u_1: E_0 \xrightarrow{\text{appeared in}} \text{Sergeant York} \]
\[ u_2: E_0 \xrightarrow{\text{married to}} E_1 \]
\[ u_3: E_1 \xrightarrow{\text{appeared in}} \text{Bewitched} \]

证据图可表示为：

text
                  married to
      Gig Young ───────────────→ Elizabeth Montgomery
          │                              │
          │ appeared in                  │ appeared in
          ↓                              ↓
   Sergeant York                      Bewitched

这些标准实体和关系只在训练后验验证中使用，Agent 在 rollout 时看不到它们。

4. STAMP 的四阶段流水线

阶段 A：完成正常 Agent rollout

Agent 使用两个工具：

json
{
  "name": "search",
  "description": "Search the web and return URLs with snippets."
}
json
{
  "name": "open",
  "description": "Open a URL and return extracted page content."
}

轨迹形式是：

text
<think>...</think>
<tool_call>search(...)</tool_call>
<tool_response>...</tool_response>
<think>...</think>
<tool_call>open(...)</tool_call>
<tool_response>...</tool_response>
...
<final answer with citations>

阶段 B：验证最终引用文档

验证 LLM 并不直接评价“哪个搜索动作写得好”，而是判断：

  • 哪个引用网页支持目标实体；
  • 哪个引用网页支持目标关系。

阶段 C：沿 URL 来源回溯首次暴露步骤

程序读取完整轨迹日志，找到每个有效支持文档的 URL 首次出现在哪一步。

阶段 D：在 GRPO 优势层进行调制

不改变原始轨迹奖励和组内排序，只改变一条轨迹内部不同步骤的更新强度。

5. 证据验证：LLM 负责什么，不负责什么

5.1 最终引用集合

设最终回答中引用、且能够和轨迹中实际观察 URL 对齐的文档集合为：

\[ C(\tau) \]

无法和轨迹日志匹配的虚构引用会被丢弃。

5.2 实体支持集合

对于目标实体 \(A_i\)0，定义：

\[ D(e)=\{d\in C(\tau):\operatorname{sup}(e,d)=1\} \]

其中 \(A_i\)1 由 LLM verifier 判断：文档 \(A_i\)2 是否真的支持实体 \(A_i\)3 的身份。

5.3 关系支持集合

对于目标关系单元 \(A_i\)4，定义：

\[ D(u_k)=\{d\in C(\tau):\operatorname{sup}(c_{ij},d)=1\} \]

关系只有在两个端点实体和关系事实都得到支持时才算验证成功：

\[ \operatorname{verified}(u_k) = \operatorname{cov}(e_i)\operatorname{cov}(e_j)\operatorname{cov}(c_{ij}) \]

5.4 LLM 不做“步骤重要性打分”

STAMP 的 verifier 不会直接输出：

text
Step 3 很关键，奖励 0.08。

它输出的是：

json
{
  "E0": {
    "correct": true,
    "supported": true,
    "source_url": "URL A"
  },
  "U2": {
    "supported": true,
    "source_url": "URL A"
  }
}

“关键步骤”是随后由确定性的 URL 首次暴露规则推导出来的。

6. 文档级信用：每个有效证据单元贡献固定预算

论文设置每个实体或关系单元的信用量为：

\[ \delta=0.03 \]

6.1 实体信用

若实体 \(A_i\)5 由多个文档共同支持，信用在这些文档之间平均分配：

\[ G=(V,E) \]0

6.2 关系信用

\[ G=(V,E) \]1

一篇文档如果同时支持实体和关系，可以从两个通道都获得信用。

这种平均分配不学习网页权重，优点是可复现；缺点是没有区分来源权威性、信息完整性和重复转载。

7. First Exposure：如何找到证据第一次被检索出来的动作

设 \(A_i\)6 表示在动作步骤 \(A_i\)7 首次对 Agent 可用的 URL 集合。对于一个支持文档 \(A_i\)8：

\[ G=(V,E) \]2

所有分给文档 \(A_i\)9 的信用，都转移到 \(c_{i,t}\)0。以后再次遇到同一 URL，不重复计分。

为什么优先奖励 discovery 而不是 read

如果轨迹为：

text
Step 1: search(...) → URL A 首次出现
Step 2: open(URL A) → 读取正文
Step 4: search(...) → URL A 再次出现

STAMP 把 URL A 的信用归给 Step 1。

作者的核心假设是：在 Deep Search 中，最困难的是构造能把关键网页“搜出来”的 query。因此，它偏向奖励 discovery action。

这是一个明确的归纳偏置，并不意味着 Step 1 已经真正理解了网页。对 DeepResearch，更精细的版本可以区分：

  • 首次 URL 暴露；
  • 首次有效段落读取；
  • 首次 claim 验证；
  • 首次冲突消解。

8. 步骤信用与上限

设步骤 \(c_{i,t}\)1 收到的实体信用总和为 \(c_{i,t}\)2，关系信用总和为 \(c_{i,t}\)3，则：

\[ G=(V,E) \]3

论文设置：

\[ G=(V,E) \]4

信用上限防止一个宽泛 query 一次返回大量相关 URL，从而让某一步的梯度过度主导训练。

整条轨迹的信用预算受证据图大小约束，而不是轨迹长度约束：

\[ G=(V,E) \]5

因此，盲目增加搜索步骤不会凭空积累更多信用。

9. GRPO 中如何注入信用

9.1 普通 GRPO

同一个问题采样 \(c_{i,t}\)4 条 rollout，奖励为 \(c_{i,t}\)5，组相对优势为：

\[ G=(V,E) \]6

普通 GRPO 将同一个 \(c_{i,t}\)6 广播给轨迹 \(c_{i,t}\)7 中所有模型动作 token。

9.2 为什么不能直接把步骤信用加到轨迹奖励

如果定义：

\[ G=(V,E) \]7

那么经过组归一化后，\(c_{i,t}\)8 仍会变成一个轨迹级优势，再次广播给所有步骤，时间定位信息会消失。

9.3 保号优势调制

STAMP 定义：

\[ G=(V,E) \]8

于是：

\[ G=(V,E) \]9

若 token \(c_{i,t}\)9 属于动作步骤 \(A^*_{i,m}\)0，则：

\[ u_1: E_0 \xrightarrow{\text{appeared in}} \text{Sergeant York} \]0

正优势轨迹

\[ u_1: E_0 \xrightarrow{\text{appeared in}} \text{Sergeant York} \]1

有效证据步骤获得更强正向强化。

负优势轨迹

\[ u_1: E_0 \xrightarrow{\text{appeared in}} \text{Sergeant York} \]2

仍然保持负号，但有效证据步骤的惩罚更弱。

保号性质

\[ u_1: E_0 \xrightarrow{\text{appeared in}} \text{Sergeant York} \]3

因此，STAMP 不会因为局部步骤有用，就把整体失败轨迹中的该步骤强行改成正样本。

11. <think>、工具调用与 token mask

STAMP 没有单独评价 <think> 内容是否正确，也没有显式的：

\[ u_2: E_0 \xrightarrow{\text{married to}} E_1 \]4

但论文定义的是 action-step 级优势。一个典型 assistant step 为：

text
<think>
需要同时利用电影和婚姻条件进一步搜索。
</think>
<tool_call>
{"name":"search","arguments":{"query":"..."}}
</tool_call>

按照论文数学描述，同一个 action step 中的模型生成 token 共享同一个 \(A^*_{i,m}\)1。通常包括：

  • <think> 中的推理 token；
  • 工具名称 token；
  • query 参数 token；
  • 工具调用格式 token。

不参与策略梯度的通常包括：

  • 用户输入；
  • 搜索引擎返回；
  • 网页正文 observation；
  • padding。

论文没有公开代码级 token mask，因此无法确认其对 <think> 是否设置了额外特殊规则。参考实现按照“所有 policy-generated action token 共享步骤优势”处理。

12. E-GRPO、C-GRPO 与 STAMP 的关系

12.1 E-GRPO

E-GRPO 使用实体命中信息给整条轨迹更细的奖励，区分：

  • 完全没找到目标实体；
  • 找到部分目标实体但最终失败；
  • 完整成功。

其本质仍然是生成一个轨迹标量：

\[ u_2: E_0 \xrightarrow{\text{married to}} E_1 \]5

12.2 C-GRPO

C-GRPO 使用答案正确性和引用 rubric 评分，对完整证据链、引用支持和约束满足情况进行轨迹级打分：

\[ u_2: E_0 \xrightarrow{\text{married to}} E_1 \]6

12.3 STAMP

STAMP 不替代上述 reward，而是接在其后：

\[ u_2: E_0 \xrightarrow{\text{married to}} E_1 \]7

所以实验结构是：

text
SFT checkpoint → GRPO + STAMP
SFT checkpoint → E-GRPO + STAMP
SFT checkpoint → C-GRPO + STAMP

而不是：

text
SFT → 完整 GRPO → 再做一轮独立 STAMP RL

STAMP 从 RL 的第一轮更新开始就嵌入优势计算过程。

13. 参考源码：核心信用计算

完整代码见：code/stamp_reference_impl.py。

13.1 文档信用

python
from collections import defaultdict


def allocate_document_credit(units, delta=0.03):
    document_credit = defaultdict(float)

    for unit in units:
        if not unit.verified or not unit.supporting_urls:
            continue

        share = delta / len(unit.supporting_urls)
        for url in unit.supporting_urls:
            document_credit[url] += share

    return dict(document_credit)

13.2 首次暴露归因

python
def first_exposure_map(steps):
    first_seen = {}

    for step in sorted(steps, key=lambda x: x.step_id):
        for url in step.exposed_urls:
            first_seen.setdefault(canonicalize_url(url), step.step_id)

    return first_seen

13.3 步骤信用

python
def attribute_credit_to_steps(document_credit, exposure_map, cap=0.10):
    raw_step_credit = defaultdict(float)

    for url, credit in document_credit.items():
        url = canonicalize_url(url)
        if url not in exposure_map:
            continue
        raw_step_credit[exposure_map[url]] += credit

    return {
        step_id: min(cap, credit)
        for step_id, credit in raw_step_credit.items()
    }

13.4 保号优势调制

python
def sign_preserving_modulation(A, c, cap=0.10):
    if not 0 <= c <= cap < 1:
        raise ValueError("Require 0 <= c <= cap < 1")

    sign = 1.0 if A > 0 else -1.0 if A < 0 else 0.0
    return A * (1.0 + sign * c)

14. PyTorch/GRPO 集成骨架

下面的输入形状为：

  • trajectory_advantages: [B]；
  • token_step_ids: [B, L]；
  • step_credit_matrix: [B, S]；
  • action_mask: [B, L]。
python
import torch


def stamp_token_advantages(
    trajectory_advantages,
    token_step_ids,
    step_credit_matrix,
    action_mask,
):
    safe_step_ids = token_step_ids.clamp(
        min=0,
        max=step_credit_matrix.shape[1] - 1,
    )

    token_credit = step_credit_matrix.gather(
        dim=1,
        index=safe_step_ids,
    )

    A = trajectory_advantages[:, None]
    A_star = A * (1.0 + torch.sign(A) * token_credit)

    # 工具返回、用户输入、padding 等位置置零
    return A_star * action_mask.to(A_star.dtype)

随后在 PPO/GRPO clipped objective 中，用 A_star 替换普通广播优势：

\[ u_2: E_0 \xrightarrow{\text{married to}} E_1 \]8

其中：

\[ u_2: E_0 \xrightarrow{\text{married to}} E_1 \]9

STAMP 不改变 PPO/GRPO clipping，只替换 advantage tensor。

15. 完整训练伪代码

python
for batch in rl_dataloader:
    # 1. 每个问题采样多个搜索轨迹
    trajectories = rollout_agent(
        policy,
        batch.questions,
        num_rollouts=8,
    )

    base_rewards = []
    all_step_credits = []

    for traj, graph in zip(trajectories, batch.evidence_graphs):
        # 2. 轨迹级奖励：GRPO / E-GRPO / C-GRPO
        reward = compute_base_reward(traj, graph)
        base_rewards.append(reward)

        # 3. 只验证最终回答真正引用的文档
        cited_docs = resolve_cited_documents(traj)
        verifier_output = verify_with_llm(
            question=traj.question,
            evidence_graph=graph,
            final_answer=traj.final_answer,
            cited_documents=cited_docs,
        )

        # 4. 实体/关系单元 → 文档信用
        document_credit = allocate_document_credit(
            verifier_output.units,
            delta=0.03,
        )

        # 5. 文档 → URL 首次暴露步骤
        exposure_map = first_exposure_map(traj.action_steps)
        step_credit = attribute_credit_to_steps(
            document_credit,
            exposure_map,
            cap=0.10,
        )
        all_step_credits.append(step_credit)

    # 6. 普通 GRPO 组内归一化
    trajectory_advantages = group_normalize(base_rewards)

    # 7. 将步骤信用广播到对应 action token
    token_advantages = build_stamp_token_advantages(
        trajectory_advantages,
        all_step_credits,
        trajectories,
    )

    # 8. 保持原有 PPO/GRPO clipped objective
    loss = grpo_loss(
        policy,
        trajectories,
        token_advantages,
    )
    loss.backward()
    optimizer.step()

19. 论文优势

19.1 不需要通用 Process Reward Model

开放网页动作空间巨大且持续变化，直接让 PRM 判断任意中间动作非常困难。STAMP 将问题转成更可验证的：

文档是否支持标准实体或关系？

19.2 不改变轨迹排序

终局 reward、组内均值和方差不变，STAMP 只缩放步骤级梯度，不会让失败轨迹反转成成功监督。

19.3 可以利用失败轨迹中的局部有效动作

这是长轨迹 Agent RL 中非常重要的能力。

19.4 与基础 reward 正交

可附加到 outcome、entity-aware、citation-aware reward 上。

20. 局限与值得继续研究的问题

20.1 URL 首次出现不等于信息首次被理解

搜索结果可能只显示标题和 URL，真正证据直到 open 后才出现，但信用仍可能归给搜索动作。

20.2 只处理最终引用文档

Agent 中途找到并使用了关键网页，但最终忘记引用，该动作不会获得 STAMP 信用。

20.3 文档级粒度仍然较粗

没有细化到：

  • 支持 claim 的具体段落；
  • 搜索 snippet 和网页正文的不同贡献；
  • 证据的权威性和独立性；
  • 复制转载与原始来源。

20.4 Step 内部仍然统一加权

关键 query 和冗长 <think> 可能共享同一优势。更细的方案可对：

  • reasoning token；
  • tool-name token；
  • query token；
  • URL token；
  • 格式 token；

使用不同系数。

20.5 验证成本

使用 GPT-5.1 对最多 15 个引用 URL 做实体和关系验证，会带来显著 API 成本和吞吐瓶颈。

20.6 实验范围有限

  • 只验证一个 30B MoE backbone；
  • 没有报告多随机种子方差；
  • 实时网页环境具有时间漂移；
  • 没有系统评估工具调用次数和搜索成本。

21. 面向 DeepResearch 的自然扩展

STAMP 当前路径是：

text
目标实体/关系
      ↓
支持网页
      ↓
URL 首次暴露动作

DeepResearch 更适合扩展为：

text
最终报告中的原子 Claim
      ↓
支持 Claim 的证据段落
      ↓
网页与来源质量
      ↓
首次发现动作 / 首次读取动作 / 首次验证动作
      ↓
分层步骤信用

可以设计：

\[ u_3: E_1 \xrightarrow{\text{appeared in}} \text{Bewitched} \]0

其中：

  • \(A^*_{i,m}\)7：首次找到关键来源；
  • \(A^*_{i,m}\)8：首次提取支持 claim 的段落；
  • \(A^*_{i,m}\)9：首次用独立来源完成交叉验证；
  • \(V\)0：发现并消解多源冲突。

这比单纯将 STAMP 套到长报告上更有研究价值。

22. 复现检查清单

  • 每个训练问题有标准实体和原子关系图；
  • 轨迹完整保存每步 action、observation、URL；
  • URL 规范化规则固定；
  • 只验证最终可解析引用；
  • verifier 输出实体/关系对应支持 URL；
  • 每个 evidence unit 总信用固定为 \(V\)1；
  • 同一文档信用只归给首次暴露步骤；
  • 单步信用截断到 \(V\)2；
  • 先计算原始 GRPO 组优势，再做步骤调制；
  • 仅 policy-generated action token 参与 loss；
  • 工具 observation 和 padding 使用 mask；
  • 对 GRPO、GRPO+STAMP 使用相同 SFT checkpoint；
  • 搜索工具、训练问题、rollout 数和上下文保持一致；
  • 单独统计关系验证率、实体率、非零信用步骤比例；
  • 至少比较 earliest、all-step、read-time 等归因策略。

23. 最后总结

STAMP 的核心贡献可以写成四句话：

  1. 奖励与信用分开：最终答案 reward 决定轨迹整体方向，证据来源决定局部步骤强度。
  2. LLM 验证证据，不直接评价动作：减少对通用过程奖励模型的依赖。
  3. 首次暴露归因：把有效引用网页追溯到首次带来该网页的动作。
  4. 保号优势调制：成功轨迹关键动作多奖励，失败轨迹局部正确动作少惩罚，但不翻转正负方向。

它仍然是 action-step 粒度的信用分配，而不是 query-token、claim 或证据段落级的因果归因。正因如此，STAMP 更像一个清晰、可操作的起点，而不是 DeepSearch/DeepResearch 信用分配问题的终点。
