Tree-GRPO：用树状 Rollout 重构 Agent 强化学习的信用分配

共享搜索前缀，在相同历史状态下比较不同后续。本文从树状 rollout 和优势估计讲到完整案例、实验结果，以及这套方法实际怎样训练。

常中豪2026 年 7 月 19 日
相关材料：论文 · 官方代码。训练配置同时参考论文 v3 和公开脚本；两者不一致的地方会在正文里单独标出。

最近在看 Tree-GRPO。起因很具体：在普通 GRPO 里，同一道题会独立采样多条完整轨迹，每条轨迹最终只有一个结果奖励。对于数学题，这种做法已经存在信用分配偏粗的问题；放到多轮搜索 Agent 里，问题会更明显——一次错误可能只发生在第三次搜索，但整条轨迹里的第一步、第二步和第三步却共享同一个优势值。

Tree-GRPO 没有再训练一个过程奖励模型，也没有要求人工标注“哪一步正确”。它改变的是 Rollout 的组织方式：先生成少量完整主干，再从已有轨迹的中间 Agent 步骤继续分叉。这样既能复用搜索前缀，又能在相同历史状态下比较不同后续决策。本文只围绕五个问题展开：过去 GRPO 的信用分配为什么不够细；Tree-GRPO 到底改了什么；一棵树如何被采样和训练；实验结果说明了什么；最后这套训练到底有没有 SFT、用了多少样本和资源。

1. 过去 GRPO 的信用分配问题

1.1 从一组完整回答计算相对优势

对问题 \(x\)，普通 GRPO 从旧策略 \(\pi_{\text{old}}\) 独立采样 \(G\) 条完整轨迹：

$$ H^1,H^2,\ldots,H^G \sim \pi_{\text{old}}(\cdot\mid x). $$

每条轨迹结束以后得到一个结果奖励 \(R(H^i)\)，再在同一问题的组内标准化：

$$ \hat A(H^i)= \frac{R(H^i)-\mu_R}{\sigma_R+\epsilon}. $$

这一步本身没有问题。GRPO 的优势是不用单独训练价值模型，而是让同一道题下的多个候选互相充当基线。问题出现在：这个优势值的统计对象是一条完整轨迹，而不是轨迹里的某一个 Agent 步骤。

在 ReAct 搜索 Agent 中，一条轨迹可能是：

text
Step 1  Thought → Search → Observation
Step 2  Thought → Search → Observation
Step 3  Thought → Search → Observation
Step 4  Final Answer

如果最终答案错误，整条轨迹得到负优势。训练时，Step 1、Step 2 和 Step 3 中由策略模型生成的 token 都会受到这条负优势影响。最终奖励只说明“整条路线没有成功”，却没有直接说明错误从哪里开始。

1.2 长轨迹把一个局部错误放大成整条轨迹的惩罚

考虑两条轨迹：

text
轨迹 A：识别人名 → 查出生地 → 查首都河流 → 正确
轨迹 B：识别人名 → 查出生地 → 误查国家最长河流 → 错误

它们前两步完全相同，真正的差异只发生在最后一次搜索。但链式 GRPO 把它们当作两条彼此独立的完整样本：A 的所有动作获得正信号，B 的所有动作获得负信号。于是，“识别人名”和“查出生地”同时出现在正、负轨迹里，却没有一个结构告诉优化器：这两个步骤其实不是造成差异的关键。

这类问题在多轮 Agent 中尤其严重，因为轨迹越长，最终奖励与早期动作之间的距离越远。论文把它概括为 稀疏监督（sparse supervision）：轨迹长度和工具调用不断增加，但监督仍然只有末尾的一个标量结果。

1.3 独立链式 Rollout 还会重复支付前缀成本

普通 GRPO 的另一个问题是成本。假设同一道题的四条轨迹都先搜索同一个人物，只在后面走向不同实体，那么四条轨迹仍会重复生成早期思考、重复调用搜索工具、重复读取相似观察结果。

链式采样 
问题
├─ A1 → A2 → A3 → 答案
├─ B1 → B2 → B3 → 答案
├─ C1 → C2 → C3 → 答案
└─ D1 → D2 → D3 → 答案

每条轨迹都从头执行，前缀无法复用。

树状采样 
问题
└─ A1 → A2
        ├─ A3 → 答案 1
        ├─ B3 → 答案 2
        └─ C3 → 答案 3

共享 A1、A2，只重新生成分叉后的部分。

因此，过去 GRPO 在 Agent 场景里同时存在两个相互关联的问题：

一方面，完整独立轨迹造成大量重复 Rollout；另一方面，每条完整轨迹只有一个统一优势，难以定位中间步骤的相对贡献。

Tree-GRPO 的出发点不是给奖励函数增加更多规则，而是先改变轨迹之间的结构关系。

2. Tree-GRPO 的核心思想与完整机制

2.1 树节点不是 token，而是完整 Agent 步骤

Tree-GRPO 把一个完整的 Thought–Action–Observation 三元组作为树节点：

$$ p_t=(\tau_t,\alpha_t,o_t), $$

其中 \(\tau_t\) 是思考，\(\alpha_t\) 是搜索或工具动作，\(o_t\) 是环境返回的观察。这样做的原因很直接：如果在半个搜索标签、半句话或某个任意 token 处分叉，新生成内容可能与已经发生的工具调用不一致。Agent 步骤天然给出了语义完整的状态边界。

一条轨迹写成：

$$ H=\{(\tau_0,\alpha_0,o_0),\ldots,(\tau_{T-1},\alpha_{T-1},o_{T-1})\}. $$

Tree-GRPO 不是传统 MCTS。它没有 UCB、访问次数或价值引导的节点选择；论文采用的是“先初始化、再随机扩展”的在线采样方式。

2.2 初始化、采样和扩展

对每一个训练问题，树状 Rollout 分三步完成。

第一步：初始化 \(M\) 棵树。 策略模型从问题起点独立生成 \(M\) 条完整轨迹，每条轨迹构成一棵树的主干。\(M\) 越大，初始路线越多，全局探索越广，但从头生成的成本也越高。

第二步：每棵树随机选择 \(N\) 个非答案节点。 这些节点可能位于早期、中期或临近答案的位置。早期分叉比较的是较长的后续策略，后期分叉则更接近某一个局部动作的比较。

第三步：从选中节点继续 Rollout。 系统保留从问题到该节点的全部上下文，然后只重新生成该节点之后的剩余轨迹。新后续被接回原树，成为一条新的叶子路径。

将“采样 \(N\) 个节点并扩展”重复 \(L\) 轮后，每个问题最终得到：

$$ G=M(LN+1) $$

条完整叶子轨迹。

关于 L 最准确的理解
L 是扩展轮数。第 1 轮从初始树中选 N 个节点；第 2 轮从已经扩大的整棵树中再选 N 个节点，因此可能选到上一轮新生成的节点，也可能再次选到原主干节点。每轮每棵树只新增 N 条轨迹，所以叶子数线性增长为 1+LN，而不是每个新分支都再生 N 个孩子的指数增长。

例如 \(M=1,N=2,L=3\)：

text
初始化：1 条主干
第 1 轮：新增 2 条叶子
第 2 轮：新增 2 条叶子
第 3 轮：新增 2 条叶子
最终：1 + 3×2 = 7 条叶子

三个超参数的分工可以压缩成一句话：

参数 控制什么 增大后的主要效果
\(M\) 独立主干数量 增强全局探索，但增加完整 Rollout 成本
\(N\) 每轮每棵树扩展的节点数 增加横向分支和局部比较，适合并行
\(L\) 扩展轮数 允许“分支上的分支”，但增加串行等待

论文的大多数标准实验使用 \(M=2,N=2,L=1\)。这意味着先生成两条独立主干，每棵树在一轮中选两个节点扩展，最终得到六条叶子轨迹。

2.3 树内优势和树间优势

树状 Rollout 完成后，所有叶子仍然只在最终答案处获得结果奖励。Tree-GRPO 的关键变化在于：同一个问题下的轨迹被组织成两个不同的统计层级。

树内优势（intra-tree advantage） 只使用同一棵树内的叶子计算均值和标准差：

$$ \hat A_{\text{intra}}(H^i)= \frac{R(H^i)-\mu_{\mathcal T_i}} {\sigma_{\mathcal T_i}+\epsilon}. $$

同一棵树里的轨迹共享较多前缀，因此树内比较更接近：在相似历史状态下，哪一种后续更好。

树间优势（inter-tree advantage） 把该问题下所有树的叶子放在一起：

$$ \hat A_{\text{inter}}(H^i)= \frac{R(H^i)-\mu_{\mathcal G}} {\sigma_{\mathcal G}+\epsilon}. $$

它提供一个更稳定的全局基线，判断一条完整轨迹相对于当前问题的全部候选总体表现如何。

最终优势是两者之和：

$$ \hat A_{\text{tree}}(H^i) = \hat A_{\text{intra}}(H^i) + \hat A_{\text{inter}}(H^i). $$

为什么不能只用树内优势？如果一棵树只有三条叶子，而且奖励全是 0，树内标准差为 0；即使奖励是 \(\{1,0,0\}\)，样本也很少，基线方差较大。树间优势相当于一个兜底的全局参照。

2.4 为什么它能产生隐式步骤偏好

设两条叶子共享前缀 \(P\)，之后分别走向成功后续 \(S^+\) 和失败后续 \(S^-\)：

$$ H^+=P\oplus S^+,\qquad H^-=P\oplus S^-. $$

当它们获得相反方向的优势时，策略梯度中共同前缀的项会在局部比较里相互抵消，主要差异落在分叉后的后续：

$$ \nabla\log \pi(S^+\mid P) - \nabla\log \pi(S^-\mid P). $$

论文进一步证明，在二元胜负假设下，树内 GRPO 与步骤级 DPO 具有相同的偏好梯度结构，差别主要在权重项。因此 Tree-GRPO 可以被理解为：在线生成“共享状态—优选后续—劣选后续”的步骤偏好数据，但不需要先离线构造 DPO 数据。

需要保留一个细节：论文实现中的树内优势是以整棵树的所有叶子为统计组，并不是为每一个分叉节点单独计算局部 \(Q\) 值。因此它属于“由树结构诱导的隐式步骤信用”，还不是显式节点价值回传。

3. 用一个多跳搜索案例跑完整个过程

下面使用一个教学案例，完整演示初始化、分支、结果奖励和优势计算。这个例子用于解释算法，不是论文原始样本。

问题：电影《盗梦空间》的导演出生在哪座城市？该城市所在国家的首都流经哪条著名河流？

标准答案：泰晤士河。

设置：

$$ M=2,\qquad N=2,\qquad L=1. $$

所以最终叶子数为：

$$ G=2(1\times2+1)=6. $$

奖励采用最简单的二元形式：答案正确记为 1，错误记为 0。

3.1 两条初始主干

第一条主干按正确关系链执行：

text
A1  搜索 Inception director
    → Christopher Nolan

A2  搜索 Christopher Nolan birthplace
    → London

A3  搜索 river flowing through London
    → River Thames

H11 最终答案：泰晤士河，R=1

第二条主干前两步正确，但最后把“流经首都的河流”偷换成“该国最长河流”：

text
B1  搜索 Inception director birthplace
    → Christopher Nolan was born in London

B2  搜索 capital of the United Kingdom
    → London

B3  搜索 longest river in the United Kingdom
    → River Severn

H21 最终答案：塞文河，R=0

3.2 从中间节点生成四条新分支

Tree 1 选择 \(A_1\) 和 \(A_2\) 两个节点扩展。

  • 从 \(A_1\) 后重新生成时，模型错误地查询 Nolan 的国籍，并把美国国籍当成出生地所在国家，最终回答波托马克河，得到 \(H_{12},R=0\)。
  • 从 \(A_2\) 后重新生成时，模型仍然误查英国最长河流，得到 \(H_{13},R=0\)。

Tree 2 选择 \(B_1\) 和 \(B_2\) 两个节点扩展。

  • 从 \(B_1\) 后重新生成时，模型重新确认 London 是英国首都，再查询流经 London 的河流，得到 \(H_{22},R=1\)。
  • 从 \(B_2\) 后重新生成时，只修正最后一次查询，得到 \(H_{23},R=1\)。

最终两棵树是：

Tree 1 
问题
└─ A1：导演是 Nolan
   ├─ A2：出生于 London
   │  ├─ A3：查流经 London 的河
   │  │  └─ H11 泰晤士河，R=1
   │  └─ A3b：误查英国最长河
   │     └─ H13 塞文河，R=0
   └─ A2a：误用国籍代替出生国家
      └─ H12 波托马克河，R=0
Tree 2 
问题
└─ B1：Nolan 出生于 London
   ├─ B2：确认 London 是英国首都
   │  ├─ B3：误查英国最长河
   │  │  └─ H21 塞文河，R=0
   │  └─ B3b：查流经 London 的河
   │     └─ H23 泰晤士河，R=1
   └─ B2a：重新确认国家和首都
      └─ H22 泰晤士河，R=1

六条叶子的结果为：

轨迹 所属树 关键决策 奖励
\(H_{11}\) Tree 1 按出生地—首都—河流关系搜索 1
\(H_{12}\) Tree 1 把国籍当成出生地所在国家 0
\(H_{13}\) Tree 1 把首都河流偷换成国家最长河 0
\(H_{21}\) Tree 2 查询英国最长河流 0
\(H_{22}\) Tree 2 重新确认关系并查询首都河流 1
\(H_{23}\) Tree 2 在最后一步修正搜索目标 1

3.3 手算树内、树间和最终优势

Tree 1 的奖励是 \(\{1,0,0\}\)。使用总体标准差并忽略极小稳定项：

$$ \mu_1=\frac13,\qquad \sigma_1\approx0.4714. $$

所以：

$$ \hat A_{\text{intra}}(H_{11})\approx1.414, $$
$$ \hat A_{\text{intra}}(H_{12}) = \hat A_{\text{intra}}(H_{13}) \approx-0.707. $$

Tree 2 的奖励是 \(\{0,1,1\}\)：

$$ \hat A_{\text{intra}}(H_{21})\approx-1.414, $$
$$ \hat A_{\text{intra}}(H_{22}) = \hat A_{\text{intra}}(H_{23}) \approx0.707. $$

全部六条轨迹的奖励是 \(\{1,0,0,0,1,1\}\)，全局均值为 0.5、标准差为 0.5，因此正确轨迹的树间优势为 +1，错误轨迹为 -1。

轨迹 树内优势 树间优势 最终优势
\(H_{11}\) +1.414 +1 +2.414
\(H_{12}\) -0.707 -1 -1.707
\(H_{13}\) -0.707 -1 -1.707
\(H_{21}\) -1.414 -1 -2.414
\(H_{22}\) +0.707 +1 +1.707
\(H_{23}\) +0.707 +1 +1.707

最重要的不是这几个具体数值，而是它们形成的比较关系。

在 Tree 1 中，三条轨迹都共享 \(A_1\)。树内优势在这个共享节点上的总和为：

$$ 1.414-0.707-0.707=0. $$

这说明局部比较不会把成功和失败的差异简单归因给共同拥有的 \(A_1\)。真正的偏好出现在分叉以后：查询真实出生地优于查询国籍；查询流经 London 的河流优于查询英国最长河流。

这就是 Tree-GRPO 的信用分配逻辑：它不直接给每一步一个人工分数，而是固定共同历史，让不同后续在最终结果上竞争。

3.4 这个例子节省了什么

假设一条完整轨迹需要三次搜索。普通 GRPO 独立生成六条完整轨迹需要约 18 次搜索。Tree-GRPO 先执行两条三步主干，需要 6 次搜索；四条分支只生成剩余后缀，在这个例子中合计约 6 次搜索，总计约 12 次。

text
普通链式采样：约 18 次搜索
Tree-GRPO：   约 12 次搜索

真实节省比例取决于分叉深度。越靠后的节点，复用前缀越长，新分支越便宜；越靠前的节点，探索范围更大，但节省更少。

4. 实验结果与消融：它究竟改善了什么

4.1 三类任务和十一项测试

论文覆盖三类问题：单跳问答、多跳问答和开放网页 Agent 问答。单跳、多跳实验使用本地 Wikipedia 检索环境；网页任务使用真实 Bing 搜索接口。单跳和多跳使用精确匹配分数，网页任务使用 F1。

最明显的提升出现在多跳问答。下面列出不同模型在四个多跳测试集上的平均分：

模型 链式 GRPO Tree-GRPO 相对提升
Qwen2.5-1.5B 11.3 19.1 +69%
Qwen2.5-3B 31.8 36.8 +16%
Llama3.2-3B 26.7 36.8 +38%
Qwen2.5-7B 36.4 37.8 +3.9%
Qwen2.5-14B 41.8 45.3 +8.4%

小模型提升尤其明显。论文观察到，链式 GRPO 在 Qwen2.5-1.5B 上很难稳定激发多轮工具使用，而 Tree-GRPO 可以通过更密集的分支比较提供更强学习信号。

单跳任务的提升较小。例如 Qwen2.5-3B 的单跳平均分从 48.1 提升到 50.0。原因并不神秘：大多数单跳问题只需要一次检索和一次回答，树深通常只有 2，几乎没有足够多的中间决策供树结构发挥作用。

4.2 同样预算下，树状采样更有效

论文将预算近似表示为“每个 prompt 相当于多少条完整轨迹的 token / 工具调用成本”。Qwen2.5-3B 的多跳结果如下：

近似预算 链式 GRPO Tree-GRPO 配置 Tree-GRPO
2 14.9 \(M=1,N=2,L=1\) 31.6
4 31.8 \(M=2,N=2,L=1\) 36.8
8 31.4 \(M=4,N=2,L=1\) 35.1
8 31.4 \(M=2,N=6,L=1\) 36.4

预算约为 2 的 Tree-GRPO 得到 31.6，已经略高于预算约为 8 的链式 GRPO 31.4。论文据此强调：对于这组设置，Tree-GRPO 使用约四分之一 Rollout 预算即可达到或超过链式基线。

这里真正起作用的不是“树越大越好”，而是预算被重新分配：少量完整主干负责探索，大量后缀分支负责在已有状态附近做有针对性的比较。

4.3 树内和树间优势的消融

优势估计的消融结果很关键。预算约为 4，配置 \(M=2,N=2,L=1\) 时：

优势方式 多跳平均分
链式 GRPO 31.8
仅树内优势 训练坍塌
仅树间优势 33.8
树内 + 树间 36.8

当分支增加到 \(N=6\) 后，仅树内优势不再坍塌，平均分达到 34.6；但树内与树间相加仍然达到最高的 36.4。

这个结果说明：

  • 树内优势提供细粒度分支偏好，但小树上的均值和方差估计容易不稳定；
  • 树间优势本身就能因为更高效的树状采样而优于链式 GRPO；
  • 两者结合，既保留局部步骤偏好，也保留稳定的全局基线。

4.4 Web Agent 的结果为什么没有多跳问答那么强

在网页任务上，Tree-GRPO 仍普遍优于链式 GRPO，但绝对提升较小。以 Qwen2.5-14B 为例：GAIA 平均分从 16.4 提升到 21.0，BrowseComp 只从 2.4 提升到 2.6。

论文明确指出，主要限制来自训练数据。网页 Agent 的训练集只有 2,200 条，而 BrowseComp 等测试问题可能需要几十次网页交互，训练样本的难度和覆盖范围无法与测试集匹配。因此，现有实验最有力地证明了 Tree-GRPO 在本地单跳、多跳搜索上的价值；对于极长开放网页 DeepResearch，它仍然是一个值得扩展、但证据尚不充分的方法。

5. 训练过程

前面一直在讲树怎样长出来，这一节把实验真正怎样训练说清楚。最容易混淆的地方有两个：Tree-GRPO 有没有先做一轮 Agent SFT，以及论文里的“训练样本数”到底指静态 QA 对，还是在线生成的叶子轨迹。

5.1 先说结论：论文主线没有单独的 SFT 阶段

论文和公开仓库都没有给出“先用工具轨迹做 SFT，再做 Tree-GRPO”的流水线。发布的训练脚本直接把 Qwen2.5 的基础检查点传给 Tree-GRPO，其中单跳 / 多跳示例使用 Qwen2.5-3B，Web Agent 示例使用 Qwen2.5-14B。ReAct 的交互格式通过 prompt 注入，随后直接进入在线 RL。

论文也报告了 Base / Instruct 系列模型，并把 Instruct 模型上的 ReAct 当作推理基线。这里要区分两件事：使用 Instruct 模型做 ReAct baseline，不等于 Tree-GRPO 在训练前额外跑了一轮 Agent SFT。公开实现里没有这一训练阶段，也没有离线的树状过程数据集。

所以主线可以画成：

text
基础模型 checkpoint
      ↓
ReAct prompt 定义 <think> / <search> / <information> / <answer>
      ↓
Tree rollout 在线生成主干和分支
      ↓
EM / F1 终局奖励 + 格式惩罚
      ↓
树内优势 + 树间优势
      ↓
GRPO 更新模型参数

每一步训练都重新用当前策略采样树。静态数据只提供问题和答案，Thought、query、Observation 以及分叉后的成功 / 失败路径全部在 rollout 时出现。

5.2 三套实验各用了多少静态样本

训练场景 训练数据 QA 对数量 搜索环境 奖励
单跳 QA Natural Questions train 79,168 本地 Wikipedia / E5，返回 Top-3 passages Exact Match
多跳 QA HotpotQA train 90,447 本地 Wikipedia / E5，返回 Top-3 passages Exact Match
Web Agent QA ASearcher-35K 中过滤的 2,000 条 + WebDancer 的 200 条困难题 2,200 Bing API，论文写返回 Top-10 F1

这三组数字是可抽样的问题数量，不是预先生成好的轨迹数量。单跳和多跳任务没有额外的 reasoning trace；Web Agent 的 2,200 条也只是经过答案过滤的问答对。

5.3 一个训练 step 会产生多少轨迹

默认树参数是 \(M=2,N=2,L=1\)，所以每个问题最后有：

$$ G=M(LN+1)=2(1\times2+1)=6 $$

条叶子轨迹。两条是从问题起点生成的完整主干，另外四条从中间节点续写。它们共享前缀，因此 6 条叶子的成本并不等于 6 条独立完整 rollout；论文把这套设置近似记为每个 prompt 4 条完整轨迹的 token / 工具预算。

根据论文给出的 step 和 batch，可以估算训练期间送入 sampler 的 prompt 槽位与叶子路径数量：

场景 step batch prompt 槽位（约） 默认叶子路径（约）
单跳 QA 180 512 92,160 552,960
多跳 QA 180 512 92,160 552,960
Web Agent QA 34 128 4,352 26,112

这里的“叶子路径”是按 \(6\times\text{prompt 槽位}\) 算出的配置量，不等于 55 万条完全独立的搜索。实际 token、工具调用和有效样本还会受提前作答、搜索上限以及最后一个 batch 影响。Web Agent 的 4,352 个 prompt 槽位接近 2,200 条数据跑 2 个 epoch，这和论文写的“34 steps，约 2 epochs”能够对上。

5.4 一轮 Tree-GRPO 更新具体经过哪些阶段

阶段一：抽题。 dataloader 从 QA 数据中取一个 prompt batch。单跳 / 多跳是 512，Web Agent 是 128。

阶段二：初始化主干。 当前策略从每个问题起点生成 \(M=2\) 条完整 ReAct 轨迹。搜索结果连同历史上下文保存在树节点中。

阶段三：扩展分支。 每棵树随机选 \(N=2\) 个非答案节点，从该状态继续生成后缀；扩展进行 \(L=1\) 轮。动态网页结果需要缓存，否则同一个前缀在恢复时可能看到不同页面。

阶段四：计算奖励。 单跳 / 多跳用 EM，Web Agent 用 F1。格式不符合协议时扣 0.2。这里没有额外 PRM，也没有逐步人工标签。

阶段五：估计优势。 同一棵树的叶子计算树内优势，当前问题的全部叶子计算树间优势，两者相加后送入 GRPO loss。

阶段六：更新策略。 优化器更新 actor；下一 step 再用新策略生成一批树。因此 Tree-GRPO 生成的是 on-policy 数据，而不是先离线生成几十万条树轨迹再统一训练。

5.6 资源使用：能确认的是 8 卡单机，卡型和时长没有公开

论文没有报告 GPU 型号、训练耗时或完整 GPU-hours。能从官方训练脚本确认的是：示例配置使用 N_NODES=1、n_gpus_per_node=8，也就是单节点 8 张 GPU。在没有作者补充信息前，不能把它写成 8×A100 或 8×H100。

训练侧基于 veRL，rollout 使用 vLLM。单跳 / 多跳脚本把 vLLM 的 gpu_memory_utilization 设为 0.6，Web Agent 设为 0.7；FSDP 的 parameter、gradient 和 optimizer offload 都打开，用 CPU 内存换显存空间。脚本中每卡对应的 PPO micro-batch 为 4，因此 8 卡时 micro-batch 配置为 32。

检索服务是另一块资源：本地 QA 需要 E5 Wikipedia 索引，官方 README 推荐 faiss-gpu，通常应单独启动检索服务；Web Agent 走 Bing API，并用 Redis 缓存搜索结果，节省重复分支的 API 调用。论文没有给检索服务器的 GPU 数，也没有公布 Web API 总调用量。

5.7 读完训练配置后，我会怎样复现

如果只验证方法，我会先选 Qwen2.5-3B 和 HotpotQA，按 1 节点 8 卡、\(M=2,N=2,L=1\)、180 steps 跑通。先确认三件事：共享前缀是否真的复用、树内 / 树间优势分组是否正确、Observation token 是否从 policy loss 中 mask 掉。随后再扩大 \(N\) 或换 Web Agent，而不是一开始就支付 Bing API 和长上下文成本。

Tree-GRPO 的训练路线并不复杂：它是直接在线 RL，复杂的是 rollout runtime。前缀缓存、节点恢复、检索结果一致性和分支并发决定了“树状采样节省预算”能不能在真实系统里兑现。
