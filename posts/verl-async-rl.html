我在 verl 里读异步强化学习：Group、队列与权重同步

最初只是想确认“一个 batch 是不是一个 GRPO group”，结果顺着这个问题一路读到了 TransferQueue、ReplayBuffer、局部更新和策略滞后。这篇把我的阅读路径和算例完整记下来。

常中豪2026 年 7 月 13 日 · verl main e5687fce

最近在看 verl 新版 PPO Trainer。起因很具体：配置里同时出现了 train_batch_size、ppo_mini_batch_size、rollout.n 和 parameter_sync_step，名字都像“这一轮要用多少数据”，但它们数的不是同一种东西。再加上 rollout.mode=async 和 trainer_mode=separate_async 同时存在，很容易把“异步请求”理解成“异步训练”。

我读的是 verl main 分支的 e5687fce，源码版本标记为 0.9.0.dev。下面不按配置项逐条翻译，而是从一个 prompt 进入系统开始，追到 actor 完成更新、Rollouter 收到新权重为止。这样更容易看清每个 batch 到底在哪里被切开，又在哪里重新合并。

1. 从 batch、group 和 trajectory 说起

1.1 一个 prompt 对应一个统计组

设一次从数据集取出 B 个 prompt。对第 i 个 prompt qi，策略采样 G 个回答：

𝒢i = { yi,1, yi,2, …, yi,G },   yi,j ∼ πold(· | qi)

𝒢i 是一个 GRPO group；G 在 verl 配置里对应 rollout.n。

1.2 Group 为什么必须完整

GRPO 不训练单独的 value model，而是在同一 prompt 的回答之间构造相对基线。最常见的组内标准化写法是：

Âi,j = ( ri,j − μi ) / ( σi + ε )

μi 与 σi 都只由第 i 个 prompt 的 G 个 reward 计算。

1.3 64 个 prompt、每题 8 个回答

因此，group 是 advantage 的统计边界，不是训练 batch 本身。若 train_batch_size=64、rollout.n=8，数据规模是：

Btrajectory = Bprompt × G = 64 × 8 = 512
量这个例子里的值它在数什么
train_batch_size64prompt / group 的数量
rollout.n8每个 prompt 采样的 response 数量
trajectory 总数512真正送入 reward 与策略优化的数据条数

这里还有一个对异步调度很关键的约束：可以让 prompt A 和 prompt C 先于 prompt B 进入训练，但不能随便拿 A 已完成的 3 个回答和未完成的 5 个回答去算组内 advantage。异步发生在 group 之间，组内仍需要一个完整的比较集合。

2. verl 代码里的两层 async

2.1 rollout.mode=async 控制什么

我一开始盯着 actor_rollout_ref.rollout.mode=async 看了很久，后来才发现当前 main 分支中同步 rollout 已经被删掉了。这个字段控制的是 rollout server 的调用方式，不负责安排 Trainer 与 Rollouter 是否重叠执行。

verl/workers/config/rollout.py
def __post_init__(self):
    # only async mode is supported
    if self.mode == "sync":
        raise ValueError(
            "Rollout mode 'sync' has been removed. "
            "Please set actor_rollout_ref.rollout.mode=async"
        )

2.2 真正选择训练调度的是 trainer_mode

决定训练调度方式的是另一处配置：

verl/trainer/config/ppo_trainer.yaml
trainer:
  use_v1: true
  v1:
    # sync | colocate_async | separate_async
    trainer_mode: sync

    colocate_async:
      num_warmup_batches: 1

    separate_async:
      num_warmup_batches: 1
      parameter_sync_step: 4

main_ppo.py 没有在入口处写三大段 if/else，而是从 registry 里取 Trainer 类：

verl/trainer/main_ppo.py
from verl.trainer.ppo.v1 import get_trainer_cls

trainer_cls = get_trainer_cls(
    config.trainer.v1.trainer_mode
)
trainer = trainer_cls(config=config)
trainer.init()
trainer.fit()

三个实现分别用装饰器注册为 sync、colocate_async 和 separate_async。所以判断一个 recipe 是否真的让训练与生成并行，我现在会先看 trainer.v1.trainer_mode，再看两边是否分配了独立 GPU；单看 rollout 的 mode 已经没有判断力。

2.3 为什么要把生成和训练做成流水线

设 Rollouter 每秒能产出 λr 个完整 group，Trainer 每秒能消费 λt 个 group。同步执行时，一轮时间近似是生成、训练和权重同步三段相加：

Tsync ≈ Trollout + Ttrain + Tweight-sync

流水线进入稳态以后，两端同时工作，系统吞吐由更慢的一端决定：

λasync ≈ min(λr, λt)

队列长度的变化可以写成 dN/dt = λ_r − λ_t。如果 λr 长期大于 λt，TransferQueue 会持续积压，轨迹版本也越来越旧；反过来，Trainer 就会在 ReplayBuffer 上等数据。异步系统真正要做的不是把队列塞满，而是让生产和消费速率尽量接近。

3. 从 prompt 到 ReplayBuffer

3.1 Trainer 先登记 prompt，再提交生成

V1 Trainer 不再要求“生成函数返回一个完整 TensorDict，我才继续往下走”。它先给每个 prompt 分配 uid，把 prompt 作为一条状态记录写进 TransferQueue，然后把真正的生成工作交给 AgentLoopManager。

trainer_base.py · _submit_batch_to_rollout
def _submit_batch_to_rollout(self, batch):
    tags = [{
        "is_prompt": True,
        "status": "pending",
        "global_steps": self.global_steps,
    }] * len(batch)

    tq.kv_batch_put(
        keys=list(batch["uid"]),
        partition_id="train",
        tags=tags,
    )
    self.agent_loop_manager.generate_sequences(batch)

这里的 global_steps 不只是日志步数。后面 ReplayBuffer 用它判断轨迹由哪个策略版本生成，以及这份数据相对当前模型已经旧了多少。

3.2 AgentLoop 的两层并发

AgentLoop 这一侧有两层并发。外层按 prompt 创建 background task，不等待结果返回；内层再按照 rollout.n 为同一个 prompt 创建多个 session。

agent_loop_tq.py · 两层并发
async def generate_sequences(self, batch):
    for i in range(len(batch)):
        task = asyncio.create_task(
            self._run_prompt(prompt, sampling_params, ...)
        )
        self.background_tasks.add(task)

async def _run_prompt(self, prompt, sampling_params, ...):
    await tq.async_kv_put(
        key=uid,
        partition_id="train",
        tag={"status": "running"},
    )

    tasks = [
        asyncio.create_task(
            self._run_agent_loop(..., session_id=i)
        )
        for i in range(n)
    ]
    await asyncio.gather(*tasks)
    await tq.async_kv_put(
        key=uid,
        partition_id="train",
        tag={"status": "finished"},
    )

这段实现解释了两个现象。第一，不同 prompt 的完成顺序可以与 dataloader 顺序完全不同；第二，finished 是在 asyncio.gather(*tasks) 之后写入的，也就是一个 prompt 的 n 个 session 都结束以后，group 才对训练端可见。

某一时刻的队列状态
prompt A   finished   A0 A1 A2 A3 A4 A5 A6 A7
prompt B   running    B0 B1 B2 B3 ...
prompt C   finished   C0 C1 C2 C3 C4 C5 C6 C7
prompt D   pending

ReplayBuffer 可以先取 A、C；不需要等 B。
但 A0~A7 仍作为同一个 group 被取走。

3.3 ReplayBuffer 不只是先进先出

我原先以为 ReplayBuffer 只是把完成的数据攒够一个 batch。实际读下来，它还承担了 group 边界、旧数据优先级和 off-policy 淘汰三件事。

replay_buffer.py · sample
while not self._has_enough_samples(
    global_steps, partition_id, batch_size
):
    time.sleep(self.poll_interval)
    self._sync_metadata_from_transfer_queue()

# 先取模型版本最老的 prompt，减少继续变旧的机会
sampleable_keys = sorted(
    finished_keys.union(failure_keys),
    key=lambda key: prompt_global_steps.get(key, 0),
)
selected_prompt_uids = sampleable_keys[:batch_size]

# 一个 uid 被选中后，把它下面的所有 trajectory 展开
for key, tag in self.partitions[partition_id].items():
    uid = key.split("_")[0]
    if uid in selected_prompt_uids:
        keys.append(key)
        tags.append(tag)

这里有个容易被教程略过去的实现细节：当前代码把 finished_keys 和 failure_keys 合并后采样。源码第 220 行还留着 TODO：是否应该过滤掉“某些 session 失败”的样本。也就是说，“group 必须全部成功才进入训练”并不是这版 ReplayBuffer 已经严格保证的性质。finished 表示全组正常结束，failure 则需要结合后续数据清洗和 padding 行为继续确认。

另一个值得注意的地方是采样单位。batch_size 先选 prompt uid，选中后再把这个 uid 下的 trajectory 展开。因此在这条 V1 路径里，ReplayBuffer 接收的 batch size 是 group 数，而不是展开后的 response 数。

4. 本地更新与权重同步

4.1 _step_once 做了哪些计算

trainer_base.py 的 _step_once 很适合用来核对一轮训练到底包含什么。删掉计时和可选分支后，主干大致是：

trainer_base.py · _step_once
batch = self.replay_buffer.sample(
    global_steps=self.global_steps,
    partition_id="train",
    batch_size=sample_batch_size,
)

batch = self._compute_reward_colocate(batch)
batch = self._balance_batch(batch)
batch = self._compute_old_log_prob(batch)

if self.use_reference_policy:
    batch = self._compute_ref_log_prob(batch)

if self.use_critic:
    batch = self._compute_values(batch)

batch = self._compute_advantage(batch)

if self.use_critic:
    batch = self._update_critic(batch)

batch = self._update_actor(batch)

这也是为什么 verl 的类名仍叫 PPOTrainer：底层管线兼容 PPO、GRPO 以及其他 advantage estimator，critic 和 reference policy 都是条件分支。使用 GRPO 时，组内 reward 产生 Â；进入 actor update 后，目标仍可写成 PPO 风格的 clipped surrogate。

ρi,j,t(θ) = πθ(yi,j,t | qi, yi,j,<t) / πold(yi,j,t | qi, yi,j,<t)
J(θ) = 𝔼[min(ρi,j,tÂi,j, clip(ρi,j,t, 1−ε, 1+ε)Âi,j)]

不同 recipe 还会叠加 KL、熵正则、长度归一化或 rollout correction；这里保留与异步问题直接相关的主项。

4.2 用 64 个 prompt 跑一遍 parameter_sync_step

parameter_sync_step 是我第二次读源码才真正分清的参数。它不是“多少 batch 才更新 actor”，而是一次全局 step 内做多少次本地训练更新，然后再把权重推给 Rollouter。

仍用前面的数字：

关键配置
data:
  train_batch_size: 64

actor_rollout_ref:
  rollout:
    n: 8
  actor:
    ppo_mini_batch_size: 16

trainer:
  v1:
    trainer_mode: separate_async
    separate_async:
      parameter_sync_step: 4

PPOTrainerSeparateAsync.__init__ 直接检查下面这个等式：

Btrain = Ksync × Bmini = 4 × 16 = 64
trainer_separate_async.py · 初始化约束
assert train_batch_size == (
    parameter_sync_step * ppo_mini_batch_size
)

rollout = config.actor_rollout_ref.rollout
assert rollout.nnodes > 0
assert rollout.n_gpus_per_node > 0
assert rollout.checkpoint_engine.backend != "naive"

随后，PPOTrainer.step 把 64 个 group 切成 4 份，每份 16 个 group：

trainer_base.py · step
sample_batch_size = (
    train_batch_size // self.parameter_sync_step
)

self._add_batch_to_generate()

for _ in range(self.parameter_sync_step):
    batch = self._step_once(
        iter_metrics,
        timing_raw,
        sample_batch_size,
    )

时间线上发生的是：

64 groups × 8 responses
local update 1: 16 groups × 8 = 128 trajectories → actor update
local update 2: 16 groups × 8 = 128 trajectories → actor update
local update 3: 16 groups × 8 = 128 trajectories → actor update
local update 4: 16 groups × 8 = 128 trajectories → actor update
-----------------------------------------------------------------
global step:    64 groups × 8 = 512 trajectories → sync weights

4.3 Local update、optimizer step 和 weight sync

更严谨地说，Trainer 执行了 4 次 _update_actor 调用。每次调用内部还会受 ppo_epochs、data parallel 切分和 micro-batch 影响，不能直接把“4 次 local update”写成“恰好 4 次 optimizer.step()”。默认 ppo_epochs=1 时两者比较接近，但换了 recipe 就未必成立。

直到外层 step 结束，on_step_end 才调用 standalone_checkpoint_manager.update_weights(self.global_steps)，把当前参数同步给独立的 rollout replicas。于是，“一个 batch 更新一次参数”这句话少了主语：Trainer 内部有自己的优化步，推理端看到新参数则由权重同步边界决定。

5. 吞吐提升以后，策略滞后怎么算

5.1 Behavior policy 与当前策略

同步训练中，数据大致由当前策略 πold 生成，更新时两者距离有限。分离式异步训练里，Rollouter 可能还在用版本 v，Trainer 已经更新到 v+k。更准确地写，轨迹来自 behavior policy μ：

μ = πθv,    πtrain = πθv+k,    Δv = k

此时 importance ratio 的分母应该对应真正生成 token 的 μ，而不是想当然地拿当前策略代替：

ρt(θ) = πθ(yt | st) / μ(yt | st)

这不是只停留在公式上的修正。PPOTrainerSeparateAsync 会强制打开 rollout-correction 的 bypass mode；基类随后直接把生成时保存的 rollout_log_probs 当作 old_log_probs。换句话说，这条路径的分母确实来自 behavior policy：

separate async · behavior policy log-prob
# trainer_separate_async.py
self.config.algorithm.rollout_correction.bypass_mode = True

# trainer_base.py
if bypass_recomputing_logprobs:
    data = tq.kv_batch_get(
        keys=batch.keys,
        partition_id=batch.partition_id,
        select_fields=["rollout_log_probs"],
    )
    old_log_probs = data["rollout_log_probs"]

源码旁边还写着 decoupled PPO 暂未支持。那种模式会区分 πrollout、稳定的 proximal anchor πold 和正在更新的 πθ 三个策略；当前 separate async 走的是更直接的两策略路径。

版本差距越大，ratio 越容易离开 clipping 区间，大量 token 的梯度会被裁掉；更糟时，旧策略覆盖的回答区域已经偏离当前策略，希望靠有限样本做 importance correction 也会变得不稳定。所以异步的收益不是免费的，它用一定程度的 off-policy 偏差换设备利用率。

5.2 drop 与 wait 怎样处理旧数据

verl V1 用 max_off_policy_threshold 限制轨迹跨越的模型版本数。drop 策略的判断很直接：

replay_buffer.py · 丢弃过旧样本
prompt_step = tag["global_steps"]
staleness = global_steps - prompt_step + 1

if staleness > self.max_off_policy_threshold:
    dropped_keys.append(key)
else:
    kept_keys.append(key)

配置里还提供 wait。它不是不管旧数据，而是在 pending/running group 即将达到阈值时，阻止训练继续向前冲，等这些 group 完成。drop 保吞吐但浪费生成算力，wait 保样本但可能重新引入等待，两者没有普遍更优的答案。

5.3 Partial rollout 会让一条轨迹跨版本

partial rollout 还会带来“同一条 trajectory 跨多个模型版本”的情况。Trainer 已经记录了 min_global_steps 与 max_global_steps，并计算：

span = vmax − vmin + 1
stalenessfresh = vcurrent − vmax,   stalenessworst = vcurrent − vmin

这几个指标比单看 generation latency 更有用。若吞吐提高了，但 trajectory_spans/max 和 trajectory_staleness_worst/mean 同时快速上升，说明系统只是把 GPU 空转问题换成了更严重的策略偏差。

6. 三种模式、配置方法和常见报错

6.1 Sync、colocate 和 separate 的资源差别

模式GPU 布局训练期间的 Rollout更适合什么情况
sync同一资源池等完整 batch，训练与生成串行先跑通、排查数值与奖励问题
colocate_asyncTrainer 与 Rollout 共用 GPU请求异步；采样结束后暂停生成、释放权重和 KV cacheGPU 数量有限，主要想减少 prompt 长尾
separate_async独立 rollout GPUTrainer 更新时，standalone Rollouter 仍可生成有独立资源，目标是形成训练—生成流水线

colocate_async 的源码很能说明“调度异步”和“计算并行”的差别。采样结束后它会终止未完成请求，并让 replicas sleep；当前 step 完成权重更新后再恢复 generation：

trainer_colocate_async.py
def on_sample_end(self):
    self.checkpoint_manager.abort_replicas()
    self.checkpoint_manager.sleep_replicas()

def on_step_end(self):
    self.checkpoint_manager.update_weights(
        self.global_steps
    )
    self.checkpoint_manager.resume_generation_replicas()

所以 colocate 模式可以让 prompt 以异步任务方式进入队列，也支持 partial rollout，但同一批 GPU 仍要在训练和推理之间切换。只有 separate 模式给 Rollouter 单独分配节点和 GPU 后，两种重计算才有条件真正重叠。

6.2 我会从什么配置开始试

如果目标是理解算法或排查 reward，我仍会先用 sync。异步系统里，生成顺序、模型版本和队列等待都会进入变量空间；基础 loss 都没对齐时就上异步，很难判断问题来自算法还是调度。

资源有限但长 prompt 很多时，我会先试 colocate：

colocate_async overrides
trainer.use_v1=true
trainer.v1.trainer_mode=colocate_async
trainer.v1.colocate_async.num_warmup_batches=1

有独立 rollout 节点时，再切 separate。下面只列与异步直接相关的 override；模型、数据和算法配置仍要沿用具体 recipe：

separate_async overrides
trainer.use_v1=true
trainer.v1.trainer_mode=separate_async
trainer.v1.separate_async.num_warmup_batches=1
trainer.v1.separate_async.parameter_sync_step=4

trainer.v1.sampler.max_off_policy_threshold=8
trainer.v1.sampler.max_off_policy_strategy=drop

actor_rollout_ref.rollout.nnodes=1
actor_rollout_ref.rollout.n_gpus_per_node=8
actor_rollout_ref.rollout.checkpoint_engine.backend=nccl

配置能启动只是第一步。我会同时看四组指标：Rollout 与 Trainer 的 GPU 利用率、ReplayBuffer 中 pending/running/finished 的数量、被丢弃样本数，以及 trajectory span / staleness。若队列越堆越长，优先增加 rollout 吞吐或降低提交速度；若队列很空而 Trainer 在等数据，问题多半在生成端；若 stale 样本大量被 drop，则应减小 parameter_sync_step、降低 warmup/在途量，或者提高权重同步频率。

最后补充我实际查版本时遇到的差异：PyPI/tag v0.8.0 里还没有这套 V1 trainer_mode 目录，全异步训练入口位于 verl.experimental.fully_async_policy.fully_async_main；main 的 0.9.0.dev 才把 sync、colocate_async、separate_async 统一接入 main_ppo。网上 recipe 混着抄时，最先应该核对的不是参数名，而是代码 commit。

6.3 我实际会先排查的几个错误

现象对应的源码约束先检查什么
启动时提示 batch size 不相等train_batch_size == parameter_sync_step × ppo_mini_batch_size三个量是否按 prompt group 计数
separate async 拒绝启动rollout 的 nnodes、n_gpus_per_node 必须大于 0是否真的为 Rollouter 分了独立资源
checkpoint backend 报错separate 模式不接受 naive按环境选择 NCCL、NIXL、Mooncake 等后端
Trainer 经常等数据ReplayBuffer 的完整 group 不足 sample_batch_size生成吞吐、超时率和 group failure 数
大量样本被丢弃轨迹版本跨度超过 max_off_policy_threshold同步频率、warmup 数量与队列积压

我不建议一看到 dropped sample 就把 threshold 调大。阈值变大只是让更多旧数据通过，并没有消除 behavior policy 与当前策略之间的距离。更有信息量的做法是先画出 staleness 分布，再对照 Rollout/Trainer 两端吞吐定位瓶颈。

7. 一个可运行的异步 group 示例

下面这段代码不依赖 verl，也不训练语言模型。它只复现最容易混淆的调度语义：每个 prompt 内部并发生成 G 条 trajectory；等 group 完整后才进入 ready queue；Trainer 每次消费若干完整 group。把它保存为 async_group_demo.py 可以直接运行。

async_group_demo.py
import asyncio
import random
from dataclasses import dataclass
from statistics import fmean, pstdev


GROUP_SIZE = 4
TRAIN_BATCH_GROUPS = 2
NUM_PROMPTS = 6


@dataclass
class Trajectory:
    prompt_id: int
    sample_id: int
    behavior_version: int
    reward: float
    advantage: float = 0.0


async def generate_one(
    prompt_id: int,
    sample_id: int,
    behavior_version: int,
) -> Trajectory:
    # 模拟不同 response 的生成时延
    await asyncio.sleep(random.uniform(0.05, 0.35))
    reward = round(random.uniform(0.0, 1.0), 3)
    return Trajectory(
        prompt_id=prompt_id,
        sample_id=sample_id,
        behavior_version=behavior_version,
        reward=reward,
    )


def normalize_group(group: list[Trajectory]) -> None:
    rewards = [x.reward for x in group]
    mean = fmean(rewards)
    std = max(pstdev(rewards), 1e-6)
    for trajectory in group:
        trajectory.advantage = (
            trajectory.reward - mean
        ) / std


async def rollout_group(
    prompt_id: int,
    behavior_version: int,
    ready_queue: asyncio.Queue,
) -> None:
    # 对应 AgentLoop 中一个 prompt 的 n 个 session
    tasks = [
        generate_one(prompt_id, sample_id, behavior_version)
        for sample_id in range(GROUP_SIZE)
    ]

    # gather 返回前，整个 group 不会进入 ready_queue
    group = await asyncio.gather(*tasks)
    normalize_group(group)

    await ready_queue.put(group)
    rewards = [x.reward for x in group]
    print(f"ready prompt={prompt_id}, rewards={rewards}")


async def trainer(
    ready_queue: asyncio.Queue,
    total_groups: int,
) -> None:
    consumed = 0
    current_version = 1

    while consumed < total_groups:
        # 一次拿 TRAIN_BATCH_GROUPS 个完整 group
        batch = [
            await ready_queue.get()
            for _ in range(TRAIN_BATCH_GROUPS)
        ]
        consumed += len(batch)

        trajectories = [x for group in batch for x in group]
        prompt_ids = [group[0].prompt_id for group in batch]
        behavior_versions = sorted({
            x.behavior_version for x in trajectories
        })
        staleness = current_version - max(behavior_versions)

        print(
            "train",
            f"prompts={prompt_ids}",
            f"trajectories={len(trajectories)}",
            f"behavior_versions={behavior_versions}",
            f"current_version={current_version}",
            f"staleness={staleness}",
        )

        # 这里只模拟 Trainer 完成一次本地更新
        await asyncio.sleep(0.15)
        current_version += 1

        for _ in batch:
            ready_queue.task_done()


async def main() -> None:
    random.seed(7)
    ready_queue = asyncio.Queue(maxsize=4)

    trainer_task = asyncio.create_task(
        trainer(ready_queue, total_groups=NUM_PROMPTS)
    )

    rollout_tasks = [
        rollout_group(
            prompt_id=i,
            behavior_version=0,
            ready_queue=ready_queue,
        )
        for i in range(NUM_PROMPTS)
    ]

    await asyncio.gather(*rollout_tasks)
    await ready_queue.join()
    await trainer_task


if __name__ == "__main__":
    asyncio.run(main())

7.2 这段代码与 verl 源码怎样对应

示例代码verl 中的对应角色
rollout_groupAgentLoopWorkerTQ._run_prompt
asyncio.gather(*tasks)等待一个 prompt 的 n 个 session 完成
ready_queue.put(group)TransferQueue 把 prompt 状态标记为 finished
Trainer 按 group 取 batchReplayBuffer.sample
current_versionglobal_steps / rollout 权重版本

这个最小实现故意没有加入 Ray、vLLM、FSDP 和真实 optimizer。把这些基础设施全部拿掉以后，异步训练最小的正确单元仍然是：完整 group 进入队列，Trainer 按 group 消费，生成策略版本随 trajectory 保存。

8. 回到最初的问题

verl 这套实现里，异步不是一个布尔开关，而是三个相互牵制的边界：group 何时完整，Trainer 一次向前走多少个局部更新，Rollouter 何时接收新权重。TransferQueue 解决生产者和消费者解耦，ReplayBuffer 决定哪些 group 此刻能被消费，parameter_sync_step 则决定训练策略与行为策略最多能拉开多快。

回到最初的问题：一个 batch 不是一个 GRPO group；一个 batch 由多个 prompt group 组成，每个 group 再展开为 rollout.n 条 trajectory。actor 的一次本地更新也不等于一次 rollout 权重同步。把这两个层次分开，后面的吞吐、off-policy 和参数版本问题才有统一的解释。
