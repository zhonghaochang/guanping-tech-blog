一个 Agent Search 轨迹是怎么样的？

这篇文章只做一件事：把一次搜索从问题到答案完整摊开，看看模型、工具和运行时各自做了什么。

常中豪2026 年 7 月 12 日

介绍 Agent Search 时，最常见的图是一个循环：模型给出 action，工具返回 observation，然后模型继续推理。图没有错，但真正调试 Agent 时，这张图几乎帮不上忙。我们需要看到的是：每一轮到底输入了什么、输出了什么，哪条证据让搜索方向发生了变化。

下面用一个具体问题来拆。用户上传了一张桥的照片，问：“这座桥叫什么，哪一年正式通车？”照片里能看出桥塔和一块很小的路牌，但仅靠图像无法可靠回答年份。

先看完整轨迹

这条轨迹参考了 OpenSearch-VL 附录中的 Kessock Bridge 案例，我把它改写成更接近工程日志的格式。关键不在桥，而在三次动作的顺序：先缩小视觉范围，再确认实体，最后查实体属性。

episode.jsonl
{"step": 0, "type": "question",
 "image": "bridge.jpg",
 "text": "这座桥叫什么，哪一年正式通车？"}

{"step": 1, "type": "tool_call", "name": "crop",
 "arguments": {"image_id": "img_0", "box": [420, 90, 980, 530]}}
{"step": 1, "type": "tool_result", "ok": true,
 "image_id": "img_1", "parent": "img_0", "path": "crop_001.jpg"}

{"step": 2, "type": "tool_call", "name": "image_search",
 "arguments": {"image_id": "img_1"}}
{"step": 2, "type": "tool_result", "ok": true,
 "candidates": [
   {"name": "Kessock Bridge", "score": 0.87},
   {"name": "Skye Bridge", "score": 0.41}
 ]}

{"step": 3, "type": "tool_call", "name": "text_search",
 "arguments": {"query": "Kessock Bridge official opening year"}}
{"step": 3, "type": "tool_result", "ok": true,
 "passages": [{"title": "Kessock Bridge", "text": "... opened in 1982 ..."}]}

{"step": 4, "type": "final",
 "text": "图中是 Kessock Bridge，1982 年正式通车。"}

这里有两个容易忽略的点。

第一，图片搜索只解决了“它是谁”，并没有解决“何时通车”。如果模型看到第一个候选就作答，答案里的年份还是参数记忆或猜测。文本搜索不是重复检索，而是在补齐另一个信息槽。

第二，裁剪结果不是一张临时图片。它有自己的 image_id，同时保存父图片和裁剪参数。后面的图片搜索引用的是 img_1。没有这层关系，跑完以后很难知道模型究竟搜了原图还是路牌。

消息历史和事件日志不是一回事

模型看到的是一串对话消息；工程侧应该保存的是事件日志。两者经常被混成一个 JSON，最后既浪费上下文，也不好排查问题。

给模型的 tool result 应该短、稳定，只保留下一步决策需要的字段：

model_messages.json
[
  {"role": "user", "content": [
    {"type": "image", "image_id": "img_0"},
    {"type": "text", "text": "这座桥叫什么，哪一年正式通车？"}
  ]},
  {"role": "assistant", "tool_call": {
    "name": "crop", "arguments": {"image_id": "img_0", "box": [420, 90, 980, 530]}
  }},
  {"role": "tool", "name": "crop", "content": [
    {"type": "image", "image_id": "img_1"}
  ]}
]

而审计日志可以更啰嗦：工具耗时、重试次数、原始 URL、缓存命中、HTTP 状态和 token 数都应该留在这里。它们对模型没用，对定位“为什么这一批 rollout 突然变慢”却很重要。

{
  "episode_id": "bridge_0042",
  "step": 2,
  "tool": "image_search",
  "latency_ms": 1834,
  "attempt": 1,
  "cache_hit": false,

  "result_count": 5,

Harness 实际上只负责四件事

模型不会直接访问搜索引擎。它输出一个动作，Harness 负责解析、执行、回填和终止。把这四件事写清楚以后，Agent 框架叫什么反而没有那么重要。

search_loop.py
def run_episode(model, tools, messages, max_steps=8):
    events = []

    for step in range(max_steps):
        raw = model.generate(messages, tool_schemas=tools.schemas())
        action = parse_one_action(raw)  # final 或恰好一个 tool_call

        if action.type == "final":
            events.append({"step": step, "type": "final", "text": action.text})
            return {"answer": action.text, "events": events, "status": "finished"}

        started = now_ms()
        try:
            result = tools.execute(action.name, action.arguments)
            observation = compact(result)  # 给模型的短结果
            ok = True
        except ToolError as exc:
            observation = {"ok": False, "error": exc.code, "retryable": exc.retryable}
            ok = False

        messages.extend([
            {"role": "assistant", "tool_call": action.as_dict()},
            {"role": "tool", "name": action.name, "content": observation},
        ])
        events.append({
            "step": step, "type": "tool", "name": action.name,
            "ok": ok, "latency_ms": now_ms() - started,
            "arguments": action.arguments,
        })

    return {"answer": None, "events": events, "status": "budget_exhausted"}

parse_one_action 很值得单独写测试。很多看起来像“模型不会用工具”的问题，其实是协议边界不清：同一轮输出两个调用、JSON 后面又接一段自然语言，或者参数名和 schema 差一个字符。

怎样判断一条轨迹好不好

只看答案是否正确是不够的。上面这条轨迹至少还要检查下面几件事：

  • 动作有没有带来新信息。连续两次搜索返回相同候选，通常说明 query 没有收窄。
  • 证据是否覆盖答案里的每个事实。“桥名”和“年份”分别需要证据，不能拿一条相似图片同时证明两件事。
  • 是否及时停止。已有桥名和官方年份后再搜第三个网站，只会增加成本和噪声。
  • 失败能否恢复。一次超时可以重试或换 query；连续格式错误则应该结束，不能让模型在坏状态里继续生成。
日志现象常见原因我会先查什么
总是直接回答训练题可被参数记忆解决，或搜索成本惩罚过重search-required 数据比例与 reward
总搜整张图没有派生图片状态，或局部搜索示例太少image_id 链和 crop 轨迹
工具越调越多缺少停止示例，最终奖励不关心成本正确轨迹的平均步数与重复 query
偶发错误拖垮整条轨迹没有区分可恢复错误与 fatal error错误码、连续失败阈值和 token mask

这条轨迹到了训练阶段会变成什么

做 SFT 时，完整轨迹会作为多轮样本送入模型，但通常只监督 assistant 生成的工具调用和最终答案。用户问题、工具返回的文字以及新图片都是条件，不应该让语言模型去“复述”。

loss_mask.py
IGNORE = -100

def labels_for(turn, token_ids):
    if turn.role == "assistant":
        return token_ids
    return [IGNORE] * len(token_ids)  # system / user / tool 全部只作条件

做 RL 时，起点更短：只有问题、图片、可用工具和可验证答案。中间轨迹由当前策略在线生成。于是同一道题会得到多条不同路线：直接猜、只做图片搜索、完整核验，或者因为格式错误提前结束。reward 决定哪类路线以后更常出现。

所以我更愿意把轨迹看成 Agent 的“可执行思路”，而不是一段很长的思维文本。它必须能被工具执行、能被日志还原，也能明确指出答案依据来自哪一步。只要这三点没有做好，后面的 SFT 或 RL 配置再漂亮，也很难知道模型究竟学到了什么。
