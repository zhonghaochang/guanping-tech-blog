DeepSearch 到底在“深”什么？

答案不在某个神秘的 <search> token 里。所谓 DeepSearch，是模型在不完整证据上反复做决定：搜什么、读什么、信什么，以及什么时候停。

01. DeepSearch 不是“更聪明的搜索框”

普通检索的输入和输出都很明确：给一个 query，返回一组文档。DeepSearch 多了一层控制。模型要根据当前证据决定下一步动作，环境执行动作，再把结果放回上下文。这个过程可能重复十几次。

因此，系统真正的状态不是一段 prompt，而是下面这组东西的合集：

  • 用户问题与原始图片；
  • 模型已经产生的推理与工具动作；
  • 每次搜索返回的候选、正文和失败信息；
  • 剩余工具预算、上下文长度和停止条件。
01Model
提出下一步动作
02Parser
校验结构与参数
03Tool
执行搜索或访问
04Observation
清洗并压缩证据
05Policy
继续、改写或停止
一个实用判断：如果代码里只有一次 `search(query)`，那是检索增强；如果模型能看完结果后重新决定动作，才开始接近 DeepSearch。

02. 先写出最小循环

下面的代码没有框架依赖。model.generate 和 tools.execute 可以换成任意模型服务和搜索实现。关键不是 API，而是控制权在谁手里。

agent_loop.py · Python
def run_agent(question, model, tools, max_turns=8):
    messages = [{"role": "user", "content": question}]
    trace = []

    for turn in range(max_turns):
        raw = model.generate(messages)
        action = parse_action(raw)
        trace.append({"turn": turn, "raw": raw, "action": action})

        if action["type"] == "answer":
            return {
                "answer": action["content"],
                "trace": trace,
                "status": "answered",
            }

        observation = tools.execute(
            name=action["name"],
            arguments=action["arguments"],
        )
        messages.extend([
            {"role": "assistant", "content": raw},
            {"role": "tool", "content": normalize(observation)},
        ])

    return {"answer": None, "trace": trace, "status": "budget_exhausted"}

这里有两个常被忽略的细节。其一，工具结果必须作为一条新消息加入，而不是拼到上一段 assistant 输出后面。其二，预算耗尽是一种明确状态，不能悄悄当成正常回答。

03. 模型何时停：EOS、stop string 与工具边界

模型不会在看到 </search> 时自动调用浏览器。它只是在生成 token。真正让生成暂停的是推理引擎或外层 Harness。

机制由谁识别适合做什么常见误区
EOS / <|im_end|>Tokenizer 与推理引擎结束当前消息把它误认为工具调用事件
Stop string推理服务遇到字符串提前截断忘记返回文本可能不含 stop 串
结构化 tool callSDK / Harness解析函数名和 JSON 参数只校验 JSON，不校验业务参数
自定义标签自写 parser训练和研究原型标签闭合后仍继续生成废话

研究代码里常见的做法，是让模型每回合只能输出一个动作：工具调用，或最终答案。Parser 应当拒绝“既调用工具又给答案”的混合输出。

protocol.py · Python
import json
import re

TOOL_RE = re.compile(r"<tool_call>(.*?)</tool_call>", re.S)
ANSWER_RE = re.compile(r"<answer>(.*?)</answer>", re.S)

def parse_action(text):
    tool_match = TOOL_RE.search(text)
    answer_match = ANSWER_RE.search(text)

    if bool(tool_match) == bool(answer_match):
        raise ValueError("one turn must contain exactly one action")

    if answer_match:
        return {"type": "answer", "content": answer_match.group(1).strip()}

    payload = json.loads(tool_match.group(1))
    if payload.get("name") not in {"text_search", "visit", "image_search"}:
        raise ValueError(f"unknown tool: {payload.get('name')}")
    return {
        "type": "tool",
        "name": payload["name"],
        "arguments": payload.get("arguments", {}),
    }

04. 工具层不要直接暴露供应商响应

搜索 API 的字段会变，网页抓取也会失败。模型不该理解每家供应商的原始 JSON。更稳妥的做法，是在工具层统一为少量稳定字段。

tools/text_search.py · Python
from dataclasses import asdict, dataclass

@dataclass
class SearchHit:
    rank: int
    title: str
    url: str
    snippet: str
    source: str

def text_search(query, provider, top_k=5):
    raw_hits = provider.search(query=query, limit=top_k)
    hits = []

    for rank, item in enumerate(raw_hits[:top_k], start=1):
        hits.append(SearchHit(
            rank=rank,
            title=item.get("title", "").strip(),
            url=item.get("url", ""),
            snippet=clean_text(item.get("snippet", ""))[:600],
            source=provider.name,
        ))

    return {
        "ok": True,
        "query": query,
        "results": [asdict(hit) for hit in hits],
    }

这段适配层做了三件事：限制 Top-K，截断每条摘要，保留来源名称。后面换 SerpAPI、Bing 或离线索引时，上层轨迹格式不必跟着变。

05. Observation 不是“把网页全塞回去”

一页正文可能有几万字。全量回注既浪费上下文，也会让导航、推荐阅读和版权声明稀释证据。Observation 应当围绕当前目标做压缩，同时保留可追溯的 URL。

observation.py · Python
def build_observation(query, hits, char_budget=4200):
    chunks = []
    used = 0

    for hit in hits:
        block = (
            f"[{hit['rank']}] {hit['title']}\n"
            f"URL: {hit['url']}\n"
            f"Evidence: {hit['snippet']}\n"
        )
        if used + len(block) > char_budget:
            break
        chunks.append(block)
        used += len(block)

    return (
        f"Search query: {query}\n"
        f"Returned: {len(hits)} results\n\n"
        + "\n".join(chunks)
    )
工程上最好同时保存两份：给模型看的压缩 observation，以及用于审计的原始 API 响应。只存前者无法复盘信息损失；只存后者则很难复现模型当时到底看到了什么。

06. 失败不是异常分支，而是训练数据

真实搜索最常见的不是“找不到答案”，而是工具半路坏掉：超时、页面 403、正文为空、图片 URL 过期、参数格式正确但语义无效。Harness 应当把失败转成结构化 observation，让模型有机会改写 query 或换工具。

tools/base.py · Python
def safe_execute(tool, arguments):
    try:
        result = tool(**arguments)
        if not result:
            return {"ok": False, "error": "empty_result", "retryable": True}
        return {"ok": True, "data": result}
    except TimeoutError:
        return {"ok": False, "error": "timeout", "retryable": True}
    except PermissionError:
        return {"ok": False, "error": "access_denied", "retryable": False}
    except Exception as exc:
        return {
            "ok": False,
            "error": type(exc).__name__,
            "retryable": False,
        }

不要把完整堆栈发给模型；那是日志，不是 observation。对策略有用的是失败类型、是否值得重试，以及本回合消耗了多少预算。

07. Prompt、SFT 和 RL 分别解决什么

Prompt 可以临时声明协议，但很难稳定长轨迹。SFT 主要解决格式、工具选择和 observation 利用；RL 才进一步优化什么时候搜、要不要继续，以及在预算内怎样取舍。

Prompt定义动作空间
SFT学会合法轨迹
RL优化搜索策略

做 SFT 时，通常只对模型产生的 token 计算 loss。用户问题与工具 observation 是条件，不是模型要背下来的答案。

loss_mask.py · Pseudocode
user(question)             loss_mask = 0
assistant(think + action)  loss_mask = 1
tool(observation)          loss_mask = 0
assistant(think + answer)  loss_mask = 1

08. 把骨架补完整

下面这份状态结构足够支撑一个小型实验。它没有包含并发搜索、图片存储和模型服务，但把后续训练真正需要的字段留了下来。

state.py · Python
from dataclasses import dataclass, field
from time import time

@dataclass
class AgentState:
    question: str
    messages: list[dict]
    trace: list[dict] = field(default_factory=list)
    tool_calls: int = 0
    max_tool_calls: int = 12
    started_at: float = field(default_factory=time)

    def can_continue(self):
        return self.tool_calls < self.max_tool_calls

    def record(self, *, raw, action, observation=None):
        self.trace.append({
            "step": len(self.trace),
            "raw_model_output": raw,
            "parsed_action": action,
            "observation": observation,
            "elapsed_ms": round((time() - self.started_at) * 1000),
        })
        if action["type"] == "tool":
            self.tool_calls += 1
到这里，DeepSearch 可以被一句话准确描述：模型负责提出动作，Harness 负责让动作发生，Observation 把现实世界带回上下文，预算与证据决定循环何时结束。
