verl 里的异步强化学习：从 Group 到权重同步

异步 RL 不是把一个开关改成 async。真正需要看清的是：谁在并发、训练何时开始、参数何时更新，以及 rollout 何时拿到新权重。

常中豪2026 年 7 月 13 日 · 基于 verl main e5687fce

大模型强化学习后训练通常有两类重计算：rollout 负责生成轨迹，Trainer 负责计算奖励、advantage 并更新 actor。同步系统按“生成一批 → 训练一批”串行执行；异步系统让已完成的轨迹先进入训练，同时 rollout 继续生产后续数据。

一句话定义：异步 RL 的核心不是“同一个 prompt 的回答没生成完就训练”，而是让完整 group 持续流入缓冲区，使 rollout 与训练形成流水线。

先统一单位：prompt、group 与 batch

在 GRPO 中，一个 prompt 会采样 n 个 response，这 n 条轨迹共同构成一个 group。组内 advantage 依赖相对奖励，因此一个 group 通常要完整生成后才能参与训练。

example.yaml
data:
  train_batch_size: 64       # 64 个 prompt group

actor_rollout_ref:
  rollout:
    n: 8                     # 每个 prompt 生成 8 条 response

这批数据包含 64 个 group、512 条 trajectory。batch 不是“一个 GRPO group 的大小”，而是一次训练阶段消费的多个 group；rollout.n 才是单个 group 的 response 数量。

verl 中有两层 async，不要混淆

actor_rollout_ref.rollout.mode=async 表示 rollout 使用异步 server 接口。当前 main 分支已经移除同步 rollout，它并不自动让 Trainer 与 Rollouter 并行。训练调度由 trainer.v1.trainer_mode 决定。

模式资源布局实际含义
sync训练与生成共享 GPU完整生成一批，再训练一批
colocate_async训练与生成共享 GPUprompt 异步调度，但 GPU 计算仍主要分阶段
separate_async训练与生成使用不同 GPU两端可同时计算，形成完整流水线

三个实现分别注册为 sync、colocate_async 和 separate_async。入口 main_ppo.py 根据配置选择 Trainer。

一条 group 在源码里怎样流动

V1 Trainer 的数据通路可以压缩成五步：

  1. Trainer 从 dataloader 取 prompt，写入 TransferQueue，初始状态为 pending。
  2. AgentLoopWorkerTQ 为每个 prompt 创建 n 个异步生成任务。
  3. asyncio.gather 等齐组内 response，再把该 group 标记为 finished。
  4. ReplayBuffer 只抽取已经完成的 group；慢 prompt 不再阻塞其他 prompt。
  5. Trainer 计算 reward、advantage，更新 critic 与 actor，再按策略向 rollout 同步权重。
conceptual_flow.txt
Prompt A ─┬─ A1 ─┐
          ├─ ... │─ group A finished ─┐
          └─ A8 ─┘                    │
Prompt B ─── still running            ├─ ReplayBuffer ─→ Trainer
Prompt C ─┬─ C1 ─┐                    │
          └─ C8 ─┘─ group C finished ─┘

关键源码是 agent_loop_tq.py 的并发生成，以及 replay_buffer.py 对完整 group 的采样。异步发生在 group 之间，GRPO 的组内比较仍然完整。

一次参数更新，不等于一次 rollout 权重同步

这是理解 separate_async 最重要的区别。假设：

separate_async.yaml
data:
  train_batch_size: 64

actor_rollout_ref:
  rollout:
    n: 8
  actor:
    ppo_mini_batch_size: 16

trainer:
  v1:
    trainer_mode: separate_async
    separate_async:
      parameter_sync_step: 4

源码要求 train_batch_size = parameter_sync_step × ppo_mini_batch_size，这里正好是 64 = 4 × 16。Trainer 每次取 16 个完整 group，更新一次 actor；连续更新 4 次后，才把新权重同步给 rollout 集群。

阶段消费数据发生的动作
本地更新 1–4每次 16 group = 128 trajectoriesTrainer actor 参数各更新一次
累计64 group = 512 trajectories共完成 4 次本地更新
阶段结束—把新权重同步给 Rollouter

这段循环位于 trainer_base.py。Trainer 更新期间，独立 Rollouter 可以继续用旧版本策略生成，因此吞吐提高，但数据不再严格 on-policy。

异步的代价：stale trajectory

假设 Rollouter 正在使用策略 v10，Trainer 已经依次更新到 v14。此时由 v10 生成、尚未消费的轨迹就是 stale trajectory。异步程度越高，设备空转越少，但策略版本差距也可能越大。

off_policy_control.yaml
trainer:
  v1:
    sampler:
      max_off_policy_threshold: 8
      max_off_policy_strategy: drop  # 或 wait

V1 sampler 可以限制可接受的版本跨度，并丢弃过旧数据。工程上要调的是三者平衡：rollout 吞吐、参数同步频率和 off-policy 偏差，而不是一味追求“越异步越好”。

怎样选择模式

  • 单机或 GPU 紧张：先用 colocate_async，它主要解决 prompt 长尾和队列调度。
  • 有独立训练与推理资源：使用 separate_async，并从较小的 parameter_sync_step 开始。
  • 只设置 rollout.mode=async：得到异步 rollout 接口，不等于异步训练流水线。
版本提示：上述 trainer.v1.trainer_mode 来自 verl main 的 0.9.0.dev 代码。verl 0.8.0 的全异步入口仍位于 verl.experimental.fully_async_policy，由 fully_async_main 同时启动 Trainer 与 Rollouter；两套配置不要混用。

结论

verl 的异步 RL 可以归结为一条清晰边界：组内 response 等齐，组间异步流动；Trainer 可以多次更新参数，Rollouter 再按设定频率接收新权重。理解这两个层次，才能正确解释 batch、group、参数更新、权重同步和 stale trajectory。
